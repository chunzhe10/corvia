# M3.4: Graph Edge Improvements — Design

> **Status:** Shipped (v0.3.0)

**Goal:** Fix the graph edge creation pipeline so that graph-expanded retrieval produces meaningfully different results from vector-only retrieval.

**Problem Statement:** The current ingestion pipeline creates 236 edges for 2,207 chunks, and nearly all edges are intra-file (same-file `contains`, `implements`, self-referential `imports`). Graph expansion during search finds 0-2 additional results per query — indistinguishable from vector-only. Three root causes were identified through benchmark investigation.

**Milestone:** M3.4 (follows M3.3 embedding/chunking pipeline)

---

## Root Causes

### 1. Broken cross-file import resolution (D72)

`resolve_rust_module_path()` in `treesitter.rs` returns the **current file path** for all `crate::` and `super::` imports instead of resolving to the target module's file. This means `use crate::config::CorviaConfig` in `main.rs` creates a self-loop (`main.rs → main.rs`) instead of a cross-file edge (`main.rs → config.rs`).

### 2. Markdown chunks have zero relations (D73)

`MarkdownChunker::chunk()` returns `relations: vec![]`. Documentation that references code files and types (via backtick code spans, links, and code blocks) creates no graph edges. ~600 markdown chunks across 37 doc files are graph-isolated.

### 3. No declaration→usage tracking across files (D74)

When `QueryableStore` is defined in `traits.rs` and used in `lite_store.rs`, `knowledge_store.rs`, and `postgres_store.rs`, there are no edges connecting them. The only cross-file relation type is `imports`, which is broken (see #1).

---

## Design

### Fix 1: Cargo.toml Module Tree Resolution (D72)

**Crate:** `corvia-adapter-git`

**Approach:** Parse `Cargo.toml` to find crate roots, scan `.rs` files for `mod foo;` declarations to build a full module→file map, then use this map in `resolve_rust_module_path()`.

**New components:**

- **`RustModuleMap`** — struct that maps module paths to file paths
  - Built by parsing `Cargo.toml` (find `[lib]`/`[[bin]]` entries, default to `src/lib.rs` or `src/main.rs`)
  - Walks each crate root scanning for `mod foo;` declarations (not inline `mod foo { }`)
  - Resolves `mod foo;` to `foo.rs` or `foo/mod.rs` by checking the file system
  - Handles workspace `Cargo.toml` with `[workspace] members` to discover all crates

- **`resolve_rust_module_path()` changes:**
  - Takes `Option<&RustModuleMap>` parameter
  - For `crate::config`, looks up the map → returns resolved file path (e.g., `crates/corvia-common/src/config.rs`)
  - For `super::foo`, resolves relative to current file's parent module
  - Falls back to current behavior if map is unavailable or path not found

- **Integration:**
  - `GitAdapter::register_chunking()` builds the `RustModuleMap` from the workspace root
  - Stores the map in `AstChunker` via `set_module_map(map)`
  - `AstChunker::chunk()` passes the map through to treesitter relation extraction

**Expected yield:** ~38 existing import relations become real cross-file edges. Cross-crate imports within the workspace (e.g., `corvia_common::config`) may also become resolvable if the workspace `Cargo.toml` is parsed.

**Alternatives rejected:**
- (A) File-system-aware resolution without Cargo.toml — simpler but can't resolve module paths correctly without knowing crate roots
- (C) Post-hoc matching in `wire_pipeline_relations()` — messier, doesn't fix the adapter's output

### Fix 2: Markdown AST Cross-References (D73)

**Crate:** `corvia-kernel`

**Approach:** Use `pulldown-cmark` to parse markdown and extract references to code files and symbols.

**New dependency:** `pulldown-cmark` added to `corvia-kernel/Cargo.toml`

**Changes to `MarkdownChunker::chunk()`:**

After building chunks, parse the full source with pulldown-cmark and extract:

1. **Code spans** (backtick content): `` `QueryableStore` ``, `` `LiteStore` ``
   - Filter to CamelCase identifiers (likely types/traits)
   - Skip short spans (≤2 chars) and Rust primitives (`String`, `Vec`, `Option`, `Result`, etc.)
   - Create `ChunkRelation { relation: "references", to_file: "", to_name: Some("QueryableStore") }`

2. **Links**: `[see traits](src/traits.rs)` or `[config](crates/corvia-common/src/config.rs)`
   - Match paths ending in known extensions (`.rs`, `.ts`, `.py`, `.toml`, etc.)
   - Create `ChunkRelation { relation: "references", to_file: "src/traits.rs", to_name: None }`

3. **Code blocks with file path references**:
   - Scan fenced code block content for file path patterns
   - Create `ChunkRelation { relation: "references", to_file: "<matched_path>", to_name: None }`

**Chunk attribution:** Map each reference to the chunk it falls in by tracking byte offsets → line numbers during parsing.

**Expected yield:** 100-200 new `"references"` edges connecting documentation to code chunks.

**Alternatives rejected:**
- (A) Regex-based extraction — simpler but more false positives, doesn't understand markdown structure
- (C) Skip markdown, rely on co-reference — misses explicit file path references in docs

### Fix 3: Declaration + Usage Tracking (D74)

**Crate:** `corvia-kernel` (in `chunking_pipeline.rs`)

**Approach:** Two-pass post-processing after all chunks are processed. First pass collects symbol definitions, second pass finds cross-file usages.

**New function:**
```
discover_cross_file_relations(chunks: &[ProcessedChunk]) -> Vec<ChunkRelation>
```

**Pass 1 — Collect definitions:**
- Scan each chunk's `metadata.chunk_type` for definition types (`"struct"`, `"trait"`, `"enum"`, `"function"`)
- Extract the declared name from chunk content (first line: `pub struct Foo`, `pub trait Bar`, `fn baz()`)
- Build definition index: `HashMap<String, Vec<(source_file, start_line)>>`
- Only track CamelCase names (types/traits) and SCREAMING_CASE (constants)
- Minimum name length: 3 chars
- Skip Rust standard library types: `String`, `Vec`, `HashMap`, `Option`, `Result`, `Box`, `Arc`, `Mutex`, `Rc`, `Cell`, `RefCell`, `Pin`, `Future`, `Stream`, `Iterator`, `Display`, `Debug`, `Clone`, `Copy`, `Send`, `Sync`, `Default`, `From`, `Into`, `TryFrom`, `TryInto`, `AsRef`, `AsMut`, `Deref`, `Drop`, `Fn`, `FnMut`, `FnOnce`, `Error`, `Ord`, `PartialOrd`, `Eq`, `PartialEq`, `Hash`, `Sized`

**Pass 2 — Find usages:**
- For each chunk, scan content for references to defined symbols
- Skip self-references (chunk defines the symbol)
- Skip same-file references (already covered by `contains`/`implements`)
- For each cross-file usage, create:
  ```
  ChunkRelation {
      from_source_file: <usage chunk's file>,
      from_start_line: <usage chunk's start line>,
      relation: "uses",
      to_file: <definition chunk's file>,
      to_name: Some(<symbol name>),
  }
  ```
- If a symbol is defined in multiple files (e.g., re-exports), link to all definitions
- If a symbol appears in 10+ files, skip it (too generic, would create edge explosion)

**Integration:**
- Called from `process_batch()` after chunks are processed
- Returns extra `ChunkRelation` entries appended to the existing relations vec
- `wire_pipeline_relations()` in CLI handles them identically

**Expected yield:** 200-400 new `"uses"` edges connecting implementations to trait/type definitions across crate boundaries.

**Alternatives rejected:**
- (A) Symbol index (co-reference) — simpler but creates undirected edges without definition/usage semantics
- (C) Embedding similarity — computationally expensive (O(n²)) and less interpretable

## Decisions

### D72: Cross-File Import Resolution via Cargo.toml Module Tree
- **Decision:** Parse `Cargo.toml` + `mod` declarations to build `RustModuleMap`, use it in `resolve_rust_module_path()`
- **Scope:** `corvia-adapter-git` crate
- **Rationale:** Current implementation returns current file for `crate::`/`super::` paths, creating self-loops. Full module tree resolution is the only way to correctly resolve Rust module paths to file paths.

### D73: Markdown AST Cross-References via pulldown-cmark
- **Decision:** Parse markdown with `pulldown-cmark` to extract code spans, links, and code blocks as `"references"` relations
- **Scope:** `corvia-kernel` crate (`MarkdownChunker`)
- **Rationale:** ~600 markdown chunks are graph-isolated. Docs heavily reference code paths and types. AST-based parsing is more precise than regex.

### D74: Declaration + Usage Tracking in Chunking Pipeline
- **Decision:** Two-pass post-processing in `discover_cross_file_relations()` to find type/trait definitions and their cross-file usages, creating `"uses"` edges
- **Scope:** `corvia-kernel` crate (`ChunkingPipeline`)
- **Rationale:** The most impactful change for graph density. When `QueryableStore` is defined in `traits.rs` and used in 4 store implementations, those edges should exist.

### D75: Lazy Search-Time Edge Discovery (deferred to M4.1)
- **Decision:** Deferred. At search time, discover missing edges and persist them for future queries. Not needed for M3.4 — ingest-time fixes will create sufficient graph density.
- **Rationale:** Adds write operations to read path, complexity not justified until ingest-time edges are validated.

## Testing Strategy

**Fix 1 tests (adapter crate):**
- `RustModuleMap` correctly parses a mini workspace with `Cargo.toml` + `mod` declarations
- `resolve_rust_module_path()` with module map resolves `crate::config` → `src/config.rs`
- `super::` resolution works relative to current file
- Full `chunk_file_with_relations()` produces cross-file import relations

**Fix 2 tests (kernel crate):**
- Markdown with `` `QueryableStore` `` produces a `"references"` relation with `to_name: "QueryableStore"`
- Markdown with `[traits](src/traits.rs)` produces a `"references"` relation with `to_file`
- Common words in backticks are filtered out
- Short spans and Rust primitives are skipped

**Fix 3 tests (kernel crate):**
- Two chunks from different files, both mentioning `QueryableStore`, produce a `"uses"` relation
- Same-file references are skipped
- Standard library types don't produce edges
- Symbols in 10+ files are skipped

**End-to-end validation:**
- Re-run benchmark script after all fixes
- Compare edge count: expect 236 → 500+ edges
- Compare benchmark results: graph expansion should show meaningfully different source file lists

## Expected Impact

| Metric | Before | After (estimated) |
|--------|--------|-------------------|
| Total edges | 236 | 500-800 |
| Cross-file edges | ~5 | 300-600 |
| Graph expansion results/query | 0-2 | 5-15 |
| Unique relation types | 3 | 5 (`imports`, `implements`, `contains`, `references`, `uses`) |
| Markdown chunks with edges | 0 | 100-200 |
