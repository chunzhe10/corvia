# M3.2: RAG Pipeline Design

**Date:** 2026-03-02
**Status:** Shipped (v0.3.0)
**Depends on:** M3 (GraphStore impl), M3.1 (GenerationEngine trait + gRPC inference)
**Decisions:** D61 (trait-based staged RAG), D62 (observability), D63 (GenerationEngine naming), D64 (token budget)

## Summary

Add a trait-based RAG pipeline to `corvia-kernel` that standardizes all retrieval, context assembly, and generation through a single orchestrated path. Three pluggable stages — `Retriever`, `Augmenter`, `GenerationEngine` — enable strategy benchmarking and future adaptation. Two modes: `context()` (retrieve + augment) and `ask()` (retrieve + augment + generate).

## Motivation

1. **No RAG today.** Corvia has strong vector retrieval but no context assembly or answer generation. MCP `corvia_search` returns raw results — agents must interpret them alone.

2. **REST/MCP bypass ContextBuilder.** Both `rest.rs:search_memories` and `mcp.rs:corvia_search` call `store.search()` directly, skipping RBAC and visibility filtering. This is a correctness bug.

3. **Graph+vector is the differentiator.** Commodity vector search is everywhere (Pinecone, pgvector, built-in IDE memory). Corvia's moat is organizational graph relationships + git-truth + temporal awareness. The RAG pipeline must combine vector similarity with graph traversal.

4. **Benchmarking readiness.** The industry moves fast. Trait-based stages let M6 eval swap in different retrieval/augmentation strategies and measure precision@k, MRR, NDCG without code changes.

## RagPipeline

```rust
pub struct RagPipeline {
    retriever: Arc<dyn Retriever>,
    augmenter: Arc<dyn Augmenter>,
    generator: Option<Arc<dyn GenerationEngine>>,
    config: RagConfig,
}

impl RagPipeline {
    /// Context-only mode: retrieve + augment, no generation.
    pub async fn context(
        &self,
        query: &str,
        scope_id: &str,
        opts: Option<RetrievalOpts>,
    ) -> Result<RagResponse>;

    /// Full RAG: retrieve + augment + generate.
    /// Returns error if GenerationEngine is not configured.
    pub async fn ask(
        &self,
        query: &str,
        scope_id: &str,
        opts: Option<RetrievalOpts>,
    ) -> Result<RagResponse>;
}
```

### Factory function

```rust
pub fn create_rag_pipeline(
    store: Arc<dyn QueryableStore>,
    engine: Arc<dyn InferenceEngine>,
    graph: Option<Arc<dyn GraphStore>>,
    generator: Option<Arc<dyn GenerationEngine>>,
    config: &CorviaConfig,
) -> RagPipeline {
    let retriever: Arc<dyn Retriever> = match graph {
        Some(g) => Arc::new(GraphExpandRetriever::new(store.clone(), engine.clone(), g)),
        None => Arc::new(VectorRetriever::new(store.clone(), engine.clone())),
    };
    let augmenter = Arc::new(StructuredAugmenter::new(config));
    RagPipeline::new(retriever, augmenter, generator, config.rag.clone())
}
```

Auto-selects `GraphExpandRetriever` when `GraphStore` is available, falls back to `VectorRetriever`.

## Retriever Implementations

### VectorRetriever

Pure vector search with visibility filtering. Equivalent to what `ContextBuilder` does today.

```
embed(query) → store.search(limit * 2) → RBAC filter → visibility filter → take(limit)
```

### GraphExpandRetriever

Vector search + graph edge expansion. Corvia's differentiator.

```
embed(query) → store.search(top-k)
    → for each result: graph.edges(entry_id, Outgoing, depth)
    → collect expanded entries via store.get()
    → deduplicate (by entry ID)
    → score: final = (1 - α) * cosine_score + α * (1.0 / (hop_distance + 1))
    → RBAC filter → visibility filter → sort by final_score → take(limit)
```

`α` is configurable (default 0.3 — vector-dominant with graph boost). Entries found only via graph get a base cosine score of 0.0 for blending.

## Augmenter Implementation

### StructuredAugmenter

Formats sources into a structured prompt with metadata and citation markers.

Output format:
```
[System prompt — configurable, has built-in default]

Context from organizational knowledge:

[1] (score: 0.94, source: src/auth.rs:45-80, type: function)
The authenticate() function validates JWT tokens...

[2] (score: 0.91, source: docs/auth-design.md, type: markdown)
Authentication uses RS256 signing...

[3] (score: 0.87, via: depends_on auth.rs, type: function)
The middleware chain calls authenticate() before...

Answer the following question using ONLY the context above.
Cite sources using [N] notation.
Question: {query}
```

Sources marked `via:` indicate graph-expanded entries (not direct vector matches).

### Token budget (D64)

Model-aware auto-sizing by default:
```
available = generator.context_window() * (1.0 - reserve_for_answer)
```

Manual override: set `max_context_tokens > 0` in config. Token estimation: chars / 4 heuristic (swappable via Augmenter trait).

Sources are packed by score until budget is exhausted. Truncated sources are counted in metrics.

## Observability (D62)

Follows D23 (OpenTelemetry stack) and D45 Part 8 (emit from day one, M4 configures export).

### Layer A: OTel Spans

```
corvia.rag.query                      — parent span
  ├─ corvia.rag.retrieve              — retrieval
  │   ├─ corvia.rag.retrieve.vector   — vector search
  │   ├─ corvia.rag.retrieve.graph    — graph expansion (if enabled)
  │   └─ corvia.rag.retrieve.filter   — visibility + RBAC
  ├─ corvia.rag.augment               — context assembly
  └─ corvia.rag.generate              — GenerationEngine call (ask mode only)
```

### Layer A: Metrics

| Type | Name | Description |
|------|------|-------------|
| Counter | `corvia.rag.queries` | Total RAG queries |
| Counter | `corvia.rag.context_only` | Queries in context-only mode |
| Histogram | `corvia.rag.retrieve.duration_ms` | Retrieval latency |
| Histogram | `corvia.rag.augment.duration_ms` | Context assembly latency |
| Histogram | `corvia.rag.generate.duration_ms` | Generation latency |
| Histogram | `corvia.rag.total.duration_ms` | End-to-end latency |
| Histogram | `corvia.rag.context.tokens` | Token count of assembled context |
| Counter | `corvia.rag.sources.included` | Entries included in context |
| Counter | `corvia.rag.sources.truncated` | Entries that exceeded budget |

### Layer A: Structured Log Events

```
rag_query_started       — INFO, query + scope_id + mode
rag_retrieved           — INFO, result_count + vector_count + graph_expanded
rag_augmented           — INFO, sources_included + token_estimate
rag_generated           — INFO, model + output_tokens
rag_query_completed     — INFO, total_latency_ms
rag_budget_exceeded     — WARN, token_estimate > budget
```

### Layer B: PipelineTrace (returned to caller)

```rust
pub struct PipelineTrace {
    pub trace_id: Uuid,
    pub retrieval: RetrievalMetrics,
    pub augmentation: AugmentationMetrics,
    pub generation: Option<GenerationMetrics>,

}

    pub latency_ms: u64,
    pub vector_results: usize,
    pub graph_expanded: usize,
    pub post_filter_count: usize,
    pub retriever_name: String,
}

pub struct AugmentationMetrics {
    pub latency_ms: u64,
    pub token_estimate: usize,
    pub token_budget: usize,
    pub sources_included: usize,
    pub sources_truncated: usize,
    pub augmenter_name: String,
}

    pub latency_ms: u64,
    pub model: String,
    pub input_tokens: usize,
    pub output_tokens: usize,
}
```

One measurement, two outputs. Timing values populate both tracing spans and the returned struct.
