## Context

Corvia v0.1.0 was a VS Code extension + Rust backend for "living documentation" — a portfolio-focused project with 10 milestones showcasing 12 LLM techniques. After further design exploration (captured in the ORM PDF), the vision expanded significantly. The "Organizational Reasoning Memory" system described in the PDF replaces the original Corvia design while keeping the name, mythology, and key principles.

### What Changed

| Aspect | v0.1.0 (Old Corvia) | v0.2.0 (New Corvia) |
|--------|---------------------|---------------------|
| Identity | Living documentation framework | Organizational reasoning memory |
| Scope | Single-repo, developer tooling | Multi-repo, domain-agnostic, general-purpose |
| Form factor | VS Code extension primary | CLI primary, VS Code as demo frontend |
| Goal | Portfolio showcase | Open source tool that's genuinely useful |
| LLM | Provider-agnostic (cloud default) | Local-first (vLLM), cloud optional |
| Storage | USearch (embedded), events.jsonl | pgvector (Docker per env), Redb (coordination) |
| Concurrency | None | Multi-agent with ephemeral namespaces, merge workers |
| Environments | Single | Isolated Docker stacks per environment set |
| Temporal model | events.jsonl + git tags | Bi-temporal (valid_time + transaction_time + environment_id) |
| Architecture | Monolithic (core + extension) | Layered kernel (kernel / adapters / frontends) |
| License | MIT | AGPL v3 (path to dual-license later) |
| Git | Core of the system | Just one use case / adapter |

## Project Identity

- **Name**: Corvia
- **Tagline**: Organizational reasoning memory for AI agents
- **Name origin**: Derived from Corvus (the raven constellation). Ravens in mythology are knowledge-gatherers — Odin's ravens Huginn (thought) and Muninn (memory) flew the world daily, gathering intelligence and returning it. "Corvia" softens the sound while preserving the lineage.
- **What it is**: A versioned, branch-aware, temporally queryable knowledge layer that multiple AI agents can read from and write to concurrently — across isolated runtime environments that it provisions and manages automatically.
- **What it is NOT**: Not a documentation tool (documentation is one use case). Not a search engine (search is one capability). Not a chatbot (chat is one frontend).
- **v1 focus**: Developer tooling — codebases, git, code understanding. The kernel is domain-agnostic but v1 is built and validated through the developer use case. Other domains (legal, clinical, financial) are architecture-level possibilities for v2+, not v1 promises. *(revised by D28)*

## Decisions Made

### D1: Relationship to Old Corvia (v0.1.0)
- **Decision**: ORM vision replaces Corvia v0.1.0 entirely
- **Alternatives considered**:
  - (A) ORM as Corvia's backend — keep VS Code frontend, swap backend for ORM architecture
  - (B) Merge best of both — cherry-pick ORM concepts into old Corvia design
  - (C) Two separate projects — Corvia for portfolio, ORM for something bigger
- **Why not A**: Treating ORM as just a backend undervalues the vision. The system IS the product, not the VS Code extension.
- **Why not B**: The ORM vision is fundamentally different in scope and philosophy. Merging would create an incoherent hybrid.
- **Why not C**: Splitting effort across two projects dilutes both. The ORM vision is strictly superior.
- **Rationale**: The ORM PDF represents the mature thinking. Old Corvia was the stepping stone that led to it. The name, mythology, and key principles carry forward; the architecture does not.

### D2: Project Name
- **Decision**: Keep "Corvia"
- **Alternatives considered**:
  - (A) New name — fresh identity for the broader vision
  - (B) Decide later — focus on design first
- **Why not A**: The raven/knowledge-gatherer mythology maps perfectly to "organizational reasoning memory." Agents gathering knowledge and bringing it back is literally the Huginn/Muninn story.
- **Why not B**: Having a name early helps with identity, repo naming, and narrative consistency.
- **Rationale**: "Corvia" is not tied to "living documentation" — it's tied to knowledge-gathering, which is exactly what the new system does.

### D3: Primary Form Factor
- **Decision**: CLI-first, VS Code extension as demo/showcase frontend
- **Alternatives considered**:
  - (A) VS Code extension primary — users interact through VS Code
  - (B) Both from the start — CLI for power users, VS Code for daily use
- **Why not A**: VS Code as primary locks the system to one IDE and one workflow. CLI is universal, scriptable, CI-friendly, and domain-agnostic.
- **Why not B**: Building both simultaneously doubles frontend work before the core is stable.
- **Why CLI-first**: The core system should work headlessly. CLI validates the kernel works. VS Code extension is layered on top as a demo that shows the system's capabilities visually — and can later connect to telemetry/observability. A strong demo lets users try the tool AND ask it questions about itself (dogfooding showcase).
- **VS Code importance**: The UI element links to telemetry later. It's a strong demo-able solution — users can try the product and ask questions about the framework through Corvia's own knowledge base.

### D4: Primary Goal
- **Decision**: Open source, genuinely useful tool. Portfolio impact is a side effect of building something real.
- **Alternatives considered**:
  - (A) Portfolio-first, commercial potential — showcase for AI engineering roles
  - (B) Commercial product — real product from day one with pricing and go-to-market
- **Why not A**: "The best portfolio is one that is useful." Optimizing for portfolio theatrics produces impressive-looking but hollow projects. Building something genuinely useful IS the portfolio.
- **Why not B**: Premature commercial focus adds constraints (pricing, support, SLAs) that slow down v1.
- **Rationale**: Open source adoption drives portfolio impact more than any contrived showcase. Good story + genuine utility = strong position.

### D5: LLM Strategy
- **Decision**: Local-first (vLLM/Ollama), cloud optional, provider-agnostic
- **Alternatives considered**:
  - (A) Fully local only — $0, no API keys, no cloud dependency ever
  - (B) Cloud-first, local optional — best results require frontier models
- **Why not A**: Locking out cloud APIs limits users who want better quality for specific tasks. Provider-agnostic is strictly more flexible.
- **Why not B**: Requiring API keys and cloud accounts creates onboarding friction for OSS. $0 default experience is critical for adoption.
- **Key distinction**: The product (Corvia) defaults to local models. The dev workflow for BUILDING Corvia uses Claude (available via subscription). Clean separation — product doesn't depend on what the developer uses.
- **Rationale**: Local-first means anyone can try Corvia with zero cost. Cloud optional means power users get better quality when they want it. Provider-agnostic means no vendor lock-in.

### D10: Core vs Complement
- **Decision**: Clear separation between the core system and complementary use cases
  - **Core**: The organizational reasoning memory primitive (knowledge store, agent coordination, embedding pipeline, environment isolation, temporal queries)
  - **Complement**: Git integration, tree-sitter chunking, VS Code extension, documentation generation, developer workflows — these are USE CASES built on the core
- **Key insight**: "Git and all is just a use case." The core must be domain-agnostic. Complements are independently valuable but don't define the system.
- **Business implication**: This separation lets you articulate the value proposition cleanly — the core is the product, the complements demonstrate it.

### D12: License
- **Decision**: AGPL v3
- **Alternatives considered**:
  - (A) MIT — Maximum permissive, zero adoption friction. Old Corvia's choice. Rejected: no SaaS protection, anyone can take the code and compete without contributing back. Closes the commercial door permanently.
  - (B) Apache 2.0 — Permissive + patent protection. Enterprise-friendly. Rejected: same lack of SaaS protection as MIT. ~5% more adoption over AGPL is not worth losing all commercial optionality.
  - (C) BSL (Business Source License) — Source-available, prevents commercial competition for 4 years. Rejected: not open source (not OSI-approved), community backlash (HashiCorp/OpenTofu), bad portfolio signal.
  - (D) Dual License from day one (AGPL + Commercial) — Requires CLA immediately. Rejected as day-one choice: slows contributions. Path to dual-license remains open for later.
  - (E) Start Apache, switch to AGPL later — Can't retroactively change license on existing versions. Community trust risk from license change.
- **Why AGPL**:
  - Fully open source (OSI-approved), not source-available pretending
  - SaaS protection: cloud providers can't offer "Managed Corvia" without open-sourcing modifications
  - Proven model: Grafana ($6B company, 7,000 enterprise customers), MinIO, Redis 8, OpenObserve all use AGPL successfully
  - Adoption impact is minimal (~5-10% vs Apache): AGPL only triggers for modifications offered as network services. Unmodified use has zero obligations.
  - Community contributions flow back: modifications to the core must be shared
  - Keeps dual-license commercial path open for later (offer commercial license to enterprises with AGPL bans)
  - Portfolio signal: "I understand open source economics" — shows business thinking alongside engineering depth
- **Key insight**: The ~5-10% of orgs with blanket AGPL bans are exactly the orgs who would pay for a commercial license if offered. AGPL barely reduces adoption while significantly protecting future options.
- **Research**: Grafana switched from Apache 2.0 to AGPL in 2021 — adoption grew, not shrank. Redis added AGPL v3 in 2025 (Redis 8). The trend in infrastructure/AI tooling is toward AGPL, not away from it.

### D11: Architecture Pattern
- **Decision**: Layered Kernel (Approach A)
- **Architecture**:
  ```
  ┌──────────────────────────────────────────────┐
  │             Frontends (complements)           │
  │  VS Code Extension │ CLI │ Web Dashboard      │
  └────────────────────┬─────────────────────────┘
                       │ MCP / gRPC
  ┌────────────────────▼─────────────────────────┐
  │           Adapters (domain use cases)         │
  │  Git/Code │ Legal Docs │ Clinical │ General   │
  │  (tree-sitter, AST)  │  (domain chunkers)    │
  └────────────────────┬─────────────────────────┘
                       │ Rust traits
  ┌────────────────────▼─────────────────────────┐
  │              Corvia Core (the kernel)         │
  │  Knowledge Store (bi-temporal, branch-aware)  │
  │  Agent Coordinator (Redb, concurrency)        │
  │  Embedding Pipeline (provider-agnostic)       │
  │  Environment Isolation (Docker lifecycle)     │
  │  Temporal Queries + Semantic Diff             │
  └──────────────────────────────────────────────┘
  ```
- **Alternatives considered**:
  - (B) Monolithic + Extension Points — ship faster with Git built-in, extension points for other domains. Rejected: Git-specific assumptions leak into core, harder to repackage for non-developer domains.
  - (C) Protocol-First (MCP-Native) — every component is an MCP server. Maximum composability. Rejected: MCP overhead (300-800ms handshake) for internal communication, over-distributed for single-machine system.
  - (D) Hybrid A+B — start monolithic, refactor to layered. Rejected: doesn't match "general from day one" — domain-specific code in core is hard to extract later.
- **Communication pattern**: Rust traits for internal hot-path (kernel ↔ adapters). MCP for external boundaries (frontends ↔ kernel, external tools ↔ kernel).
- **Rationale**: "Core vs complement" maps directly to kernel vs adapter. Dogfooding works naturally (Git adapter is just the first adapter). OSS story is clean (others can build adapters without forking). Avoids Git leakage risk.

### D13: Memory Architecture
- **Decision**: Three-store architecture (Git + pgvector + Redb) with pluggable remote storage
- **Store 1 — Git (local)**: Source of truth for all knowledge artifacts (decisions, manifests, conversation logs, configuration). Version-controlled locally via Git regardless of domain.
- **Store 2 — pgvector (Docker)**: Queryable semantic memory. One instance per environment. Auto-provisioned by Corvia — Docker is invisible to the user (`corvia init` handles Docker check, image pull, container start). Always required (no embedded fallback).
- **Store 3 — Redb (embedded Rust)**: Operational coordination state. Branch registry, agent sessions, merge queue, master environment registry. Pure Rust, no separate process, MVCC semantics.
- **Remote storage**: Pluggable via trait abstraction. Options: GitHub, GitLab, S3, local filesystem, no remote. Not tied to any single platform.
- **Memory hierarchy** (durability order): Remote → Local Git → pgvector volumes (WAL) → Redb files (WAL) → Agent in-memory (ephemeral)
- **Key principle**: Memory ultimately lives in Git. pgvector and Redb exist to make memory fast and queryable, not to be its home. `corvia rebuild` reconstructs everything from Git.
- **Crash recovery**: Inherited from ORM PDF — embedding-in-progress chunks re-queued, agent sessions buffered to Git, pgvector WAL handles unclean shutdown, Docker volumes persist across container restarts.
- **Alternatives considered**:
  - (A) Embedded vector store (USearch/SQLite-vec) for zero-Docker onboarding, pgvector for scale — Rejected: Docker is required anyway for environment isolation, so pgvector comes free. One vector store backend to maintain, not two.
  - (B) Pluggable source of truth (not necessarily Git) — Rejected: Git is the best tool for tracking text changes over time, regardless of domain. The remote is pluggable instead.
- **Rationale**: Docker dependency is accepted because environment isolation already requires it. Auto-setup makes it invisible. pgvector's MVCC and concurrent access are worth the dependency. Redb is the perfect embedded coordination store for Rust.

### D14: Temporal Data Model
- **Decision**: Bi-temporal data model + append-only event log (no CQRS). The PDF's "simplified hybrid."
- **Schema** (generalized for domain-agnostic use):
  ```yaml
  knowledge_entry:
    id: uuid
    content: "..."
    source_version: "abc123"       # domain-specific version reference
    scope_id: "project-alpha"      # isolation scope (was environment_id)
    recorded_at: 2024-03-01        # transaction time (when recorded)
    valid_from: 2024-02-15         # valid time (when true in reality)
    valid_to: null                 # still current, or timestamp when superseded
    workstream: "main"             # parallel work context (was branch)
    superseded_by: uuid | null
  ```
- **Event log**: Append-only `events.jsonl` as secondary recording for audit trail, history narration, and temporal walk-through. Current state is live — NOT derived from events.
- **Alternatives considered**:
  - (A) Single timestamp only — Rejected: can't distinguish "when true" from "when recorded." Adding valid_time later requires painful backfill of all existing entries.
  - (B) Full event sourcing + CQRS — Rejected: eventual consistency headaches, event schema evolution pain, tooling overhead, replay performance degradation. Production practitioners report teams spend more time building infrastructure than solving problems. "A history table gets 80% of the value with none of the cost."
  - (C) Bi-temporal only (no event log) — Rejected: loses audit trail narrative capability. Event log enables "walk through what happened" without the complexity of CQRS.
- **Generalized naming**: Fields renamed from ORM PDF to be domain-agnostic. `source_commit` → `source_version`, `environment_id` → `scope_id`, `branch` → `workstream`. Adapters map these to domain concepts.
- **Rationale**: Bi-temporal is not the limiting choice — it subsumes single-timestamp. Starting with it is strictly more flexible. Validated by Zep's temporal knowledge graph architecture (production AI agent memory system using bi-temporal model, published 2025). Event log provides audit trail without CQRS complexity.

### D15: Queryable Store — SurrealDB (replaces pgvector in D13)
- **Decision**: Use SurrealDB 3.0 in Docker (server mode) as the unified queryable store. Replaces pgvector. Designed behind a trait boundary so it can be swapped for a self-built solution later.
- **What changes from D13**:
  - Store 2 was: pgvector (vectors only)
  - Store 2 is now: SurrealDB (vectors + graphs + time-series + documents — all in one service)
  - Stores 1 (Git) and 3 (Redb) are unchanged
- **Updated three-store architecture**:
  - **Store 1 — Git (local)**: Source of truth. Unchanged from D13.
  - **Store 2 — SurrealDB (Docker)**: Unified queryable store. Vector search (HNSW), graph traversal (record links + graph queries), time-series, document storage. One instance per environment. Auto-provisioned.
  - **Store 3 — Redb (embedded Rust)**: Operational coordination. Unchanged from D13.
- **Why SurrealDB over pgvector**:
  - One service replaces two concerns (vectors + graphs) — simpler ops
  - Native graph capabilities enable entity-relationship queries, multi-hop reasoning, causal chains — things pgvector cannot do
  - SurrealQL is SQL-like (low learning curve vs Cypher or Datalog)
  - Pure Rust, 29K+ GitHub stars, $23M funding (Feb 2026), active development
  - MCP-based agent memory support built into 3.0
  - Time-series native — aligns with bi-temporal model (D14)
- **Why server mode (Docker), not embedded**:
  - SurrealDB core is BSL 1.1. Corvia is AGPL v3.
  - Embedded mode = static linking → AGPL requires entire combined work under AGPL-compatible terms → BSL's DBaaS restriction conflicts with GPL "freedom 0"
  - Server mode = separate process communicating via TCP → no copyleft propagation across process boundary
  - Rust SDK (the client library) is Apache 2.0 — fully AGPL-compatible
  - Docker was already required (D13 accepted Docker for environment isolation), so no new dependency
- **Trait boundary for future swap**:
  - All kernel code accesses Store 2 through a `QueryableStore` trait (name TBD)
  - Trait covers: vector search, graph queries, temporal queries, CRUD, schema management
  - SurrealDB is the first (and current) implementation
  - Future option: self-built embedded store (Rust, no Docker, no license conflict) implementing the same trait
  - Adapters and frontends never touch SurrealDB directly — only through the trait
  ```rust
  // Conceptual — not final API
  trait QueryableStore: Send + Sync {
      // Vector operations
      async fn semantic_search(&self, query: &Embedding, scope: &ScopeId, limit: usize) -> Vec<KnowledgeEntry>;

      // Graph operations
      async fn relate(&self, from: EntryId, to: EntryId, relation: &str, props: Properties) -> Result<()>;
      async fn traverse(&self, from: EntryId, depth: u32, filter: TraversalFilter) -> Vec<GraphPath>;

      // Temporal queries
      async fn as_of(&self, scope: &ScopeId, valid_at: Timestamp, recorded_at: Timestamp) -> TemporalView;
      async fn history(&self, entry: EntryId) -> Vec<TemporalSnapshot>;

      // CRUD
      async fn insert(&self, entry: KnowledgeEntry) -> Result<EntryId>;
      async fn update(&self, id: EntryId, patch: EntryPatch) -> Result<()>;
      async fn get(&self, id: EntryId) -> Option<KnowledgeEntry>;
  }
  ```
- **Migration path to self-built**:
  1. Current: SurrealDB in Docker, accessed via `QueryableStore` trait
  2. Later: Build embedded Rust store (e.g. Redb extensions, or custom B-tree + HNSW) implementing same trait
  3. Swap is a config change, not a rewrite — kernel, adapters, frontends don't change
  4. Can run both in parallel during migration (read from new, verify against old)
- **Alternatives considered**:
  - (A) pgvector (original D13) — Rejected for this decision: no graph capabilities, would need a second service for graphs later anyway
  - (B) SurrealDB embedded mode — Rejected: BSL 1.1 ↔ AGPL v3 license conflict when statically linked. Would require changing Corvia to Apache 2.0 (losing SaaS protection) or adding a linking exception (legally novel, untested)
  - (C) Kuzu (embedded graph DB) — Eliminated: Apple acquired Kuzu Oct 2025, GitHub repo archived, website down. Community forks (bighorn, Ladybug) exist but are early and uncertain. Too risky for a core dependency.
  - (D) Cozo (embedded Rust graph DB) — Rejected: solo maintainer, small community, Datalog query language (niche, harder for contributors). Risky for a core dependency despite good technical fit.
  - (E) Skip graph entirely, add later — Rejected in favor of SurrealDB: SurrealDB gives graphs "for free" alongside vectors in one Docker service. No reason to leave capability on the table when the ops cost is the same as pgvector.
  - (F) Build custom from day one — Rejected for now: significant engineering effort before core features are validated. SurrealDB buys time to learn what the graph layer actually needs in practice, then build a targeted replacement.
- **License detail**:
  - SurrealDB core: BSL 1.1 (converts to Apache 2.0 on Jan 1, 2030 for v3.0)
  - SurrealDB Rust SDK: Apache 2.0 (AGPL-compatible)
  - Communication: TCP (network boundary, not linking)
  - Corvia ships: AGPL v3 binary + Docker image reference for SurrealDB (separate artifact)
  - No license conflict in this configuration
- **Rationale**: SurrealDB gives us vectors + graphs + time-series in one Docker service at the same ops cost as pgvector alone. Server mode avoids the BSL/AGPL conflict. The trait boundary is the critical architectural choice — it makes SurrealDB a replaceable implementation detail, not a permanent dependency. When we understand our graph access patterns from real usage, we can build a targeted embedded replacement.

### D16: Concurrency Model — Multi-Agent Write Safety
- **Decision**: LLM-assisted merge as default, configurable strategy per workspace
- **How it works**:
  - Each agent gets an **ephemeral namespace** (private write space). Reads come from shared state, writes are isolated.
  - Agent sessions tracked in Redb (start, heartbeat, timeout, commit/abort).
  - On commit, writes enter a **merge queue** processed by a background merge worker.
  - Merge worker detects conflicts (two agents wrote to the same entry in the same scope during overlapping sessions) and applies the configured strategy.
  - Non-conflicting writes merge automatically (no strategy needed).
- **Strategy ladder** (configurable per workspace, ordered by intelligence):
  1. **LLM merge (default)** — an agent reviews conflicting writes, understands semantic intent, produces a merged result. Most aligned with Corvia's AI-native identity.
  2. **Merge policies** — deterministic rules for known patterns (e.g. "append-only for logs," "latest-timestamp for status fields," "higher-priority-agent-wins"). Fast, predictable, no LLM cost.
  3. **Last-write-wins** — timestamp-based, zero intelligence. Escape hatch for speed-critical or low-value conflicts.
- **Uniform output**: All strategies produce a `MergeResult` type regardless of which strategy ran. Downstream consumers never know or care which strategy resolved the conflict. This is the key to safe strategy switching later.
- **Optimistic concurrency**: Agents don't lock resources. They write freely to their namespace. Conflicts are detected at merge time, not prevented at write time. This maximizes agent throughput.
- **Risk of changing default later**: Low.
  - Strategy is a function behind the merge worker interface — swapping is a config change, not a redesign
  - The merge strategy only fires on actual conflicts (minority of writes — most agents write to different scopes/entities)
  - Uniform `MergeResult` output means downstream consumers are strategy-agnostic
  - The real coupling risk is in result shape, not strategy choice — mitigated by `MergeResult` contract
- **Alternatives considered**:
  - (A) Last-write-wins as default — Rejected: lossy, doesn't leverage Corvia's AI-native positioning. Acceptable as fallback, not as default.
  - (B) Pessimistic locking (agents acquire locks before writing) — Rejected: reduces throughput, creates deadlock risk, doesn't scale with many concurrent agents.
  - (C) Require users to choose strategy upfront — Rejected: friction at onboarding. Strong default + configuration is better UX.
- **Rationale**: An AI-native system should use AI for its own coordination. LLM merge is the distinctive default — it's what separates Corvia from systems that punt to timestamps. The strategy ladder gives users an escape hatch without forcing everyone through configuration.

### D17: Namespace Architecture
- **Decision**: Five-segment hierarchical namespace with typed struct internally, string serialization at boundaries only.
- **Format**: `{org}:{scope_id}:{workstream}:{source}:{version_ref}`
- **Version ref types**:
  - `@{hash}` — immutable (content-addressed, cacheable forever)
  - `:{label}` — mutable (resolves to immutable, re-resolved each read)
- **Examples**:
  ```
  local:project-alpha:main:my-repo:@abc123       # immutable snapshot
  local:project-alpha:main:my-repo::latest        # mutable, resolves to @abc123
  acme:env-prod:feature-x:contracts::head         # mutable, org-scoped
  ```
- **`org` default**: `local` for single-user. Multi-tenant ready without requiring it day one.
- **Struct-first implementation** (critical mitigation):
  ```rust
  struct Namespace {
      org: OrgId,
      scope: ScopeId,
      workstream: WorkstreamId,
      source: SourceId,
      version_ref: VersionRef, // enum { Immutable(Hash), Mutable(Label) }
  }
  ```
  - All internal code works with the typed struct, never raw strings
  - String format (`local:proj:main:repo:@abc`) is a serialization concern at system boundaries (API, logs, display)
  - If string format needs to change later, update the serializer — not every consumer
- **Industry validation**:
  - Hierarchical keys: proven at scale (S3, OCI registries)
  - Content-addressed immutable refs: proven (Git SHA, Docker digest, IPFS CID)
  - Mutable → immutable resolution: proven (Git refs → commits, Docker tags → digests)
  - Colon separator: used by URN (RFC 8141), Docker, Kubernetes resource paths
- **Scalability assessment**:
  - Fixed depth (5 segments) → O(1) parsing
  - Immutable refs infinitely cacheable → linear read scalability
  - Nested scopes supported via `/` within `scope_id` (e.g. `prod/team-a`)
  - Multi-source entries (edge case): handled by graph relationships (D15), not namespace
- **Change risk**: Higher than most decisions — namespaces get persisted across all three stores (SurrealDB, Redb, Git). The struct-first approach is the mitigation: internal representation can evolve independently of serialization format.
- **Alternatives considered**:
  - (A) Four-segment (no workstream) — Rejected: ambiguous when agents on different workstreams read from same scope+source+version. Workstream is a distinct concept from scope (D14).
  - (B) URI format (`corvia://{org}/{scope}/{source}?ref={ver}`) — Rejected: over-engineering for internal identifiers. Can add URI mapping at API boundary later without changing internal representation.
  - (C) Pure content-addressing (hash only, no hierarchy) — Rejected: loses human readability and hierarchical access control. Agents need context, not just content.
- **Rationale**: The scheme follows proven industry patterns (Git, OCI, URN) without inventing anything novel. Workstream inclusion fixes an ambiguity gap. The struct-first approach makes the string format a replaceable detail. Scalable technically and safe to evolve.

### D18: Cross-Source Dependencies
- **Decision**: Full cross-source support in v1 — dependency graph, knowledge promotion, pre-flight conflict detection.
- **Dependency graph**: Sources can declare dependencies on other sources. Stored as graph edges in SurrealDB (D15). Domain-agnostic — works for repos, contracts, documents, any source type.
  - Example (Git adapter): `service-a-api` → `service-b-client` (API contract dependency)
  - Example (Legal adapter): `master-agreement` → `compliance-checklist` (regulatory dependency)
- **Knowledge promotion**: Explicit, never automatic. An agent or user promotes validated knowledge from one scope to another. Prevents accidental knowledge leakage across isolation boundaries.
  - Promotion creates a new entry in the target scope with a `promoted_from` reference (graph edge back to origin)
  - Promoted knowledge is a copy, not a link — target scope owns it after promotion
- **Pre-flight conflict detection**: Before an agent commits writes, the merge worker checks if any dependent sources have conflicting knowledge. Uses the dependency graph to traverse affected sources.
  - Fails fast with a clear error: "your write to X conflicts with knowledge in dependent source Y"
  - Agent can resolve, override (with audit trail), or abort
- **Why this fits v1**: The architectural pieces already exist:
  - Namespace (D17) supports cross-source addressing
  - SurrealDB graph layer (D15) stores dependency edges naturally
  - Merge worker (D16) already processes writes — pre-flight check is an additional step in the same pipeline
  - Scope isolation is already enforced — promotion is a controlled boundary crossing
- **Alternatives considered**:
  - (A) Promotion only (no dependency graph, no pre-flight) — Rejected: covers the simplest case but misses the value of knowing which sources affect which. Without the graph, promotions are blind.
  - (B) Defer to v2 — Rejected: cross-source is part of what makes Corvia non-trivial (D8). Scoped isolation without cross-source awareness is just "multiple independent databases."
- **Rationale**: Cross-source dependencies are what distinguish an organizational memory from isolated knowledge silos. The dependency graph + promotion + pre-flight model gives structure to cross-boundary knowledge flow without sacrificing isolation. Engineering complexity is manageable because the infrastructure (graph store, merge worker, namespaces) already supports it.

### D19: Environment Isolation & Shared Infrastructure
- **Decision**: Shared SurrealDB (1 instance) with namespace-per-environment. Shared inference engine (1 pool). Isolation enforced at the data pipeline level, not the compute/storage infrastructure level.
- **Threat model clarification**: Agent isolation is about **correctness** (preventing context pollution / hallucination from neighbor data), NOT about **security** (adversarial multi-tenant access). Multi-org security isolation is a v2+ concern with a different architecture.
- **Updated architecture**:
  ```
  ┌──────────────────────────────────────────────────┐
  │                 Shared Services                   │
  │  ┌────────────────────────────────────────────┐  │
  │  │  Inference Engine (1 pool)                 │  │
  │  │  vLLM / Ollama / Dynamo / Triton           │  │
  │  │  - stateless per request                   │  │
  │  │  - embeddings + reasoning, all shared      │  │
  │  │  - InferenceEngine trait                   │  │
  │  ├────────────────────────────────────────────┤  │
  │  │  SurrealDB (1 instance, 1 Docker container)│  │
  │  │  ├── Namespace: env-a (auth-scoped, RBAC)  │  │
  │  │  ├── Namespace: env-b (auth-scoped, RBAC)  │  │
  │  │  └── Namespace: env-c (auth-scoped, RBAC)  │  │
  │  │  - QueryableStore trait (D15)              │  │
  │  ├────────────────────────────────────────────┤  │
  │  │  Redb (coordination, env-scoped keys)      │  │
  │  ├────────────────────────────────────────────┤  │
  │  │  Context Builder (isolation logic)         │  │
  │  │  - checks agent session permissions        │  │
  │  │  - queries only authorized namespace(s)    │  │
  │  │  - assembles prompt with scoped knowledge  │  │
  │  ├────────────────────────────────────────────┤  │
  │  │  Result Router (write scoping)             │  │
  │  │  - writes only to agent's own namespace    │  │
  │  └────────────────────────────────────────────┘  │
  └──────────────────────────────────────────────────┘
  ```
- **Why shared inference engine**:
  - LLM inference is stateless per request — no context retained between requests
  - Embeddings are pure functions (text → vector), safe to share by definition
  - Reasoning is stateless per prompt — the prompt is where isolation lives, not the engine
  - Cost: 1 engine vs N engines eliminates linear resource scaling with environments
  - Trait-based: `InferenceEngine` trait supports local (vLLM/Ollama) and production (Dynamo/Triton) backends
- **Why shared SurrealDB with namespaces**:
  - SurrealDB namespaces are DB-level isolation boundaries, not application-level filters
  - Connections authenticate TO a specific namespace — queries can't cross without separate credentials
  - RBAC per namespace (Owner/Editor/Viewer) prevents accidental cross-reads
  - "Not possible to move data between namespaces" (SurrealDB docs)
  - Cost: 1 instance (~200MB) vs N instances (N × ~150MB)
  - Environment creation: milliseconds (`DEFINE NAMESPACE`) vs seconds-minutes (Docker start)
  - Cross-context access is natural: open second namespace-scoped connection when permitted
- **Isolation layers** (defense in depth for correctness):
  1. **Context builder** — permission check against agent session config
  2. **SurrealDB namespace auth** — connection scoped to namespace with RBAC
  3. **Result router** — writes only to agent's own namespace
- **Cross-context access model**:
  - Default: agents read/write ONLY within their own namespace + workstream
  - Explicit grant: agent session config allows read access to specific other namespaces
  ```yaml
  agent_session:
    id: agent-42
    environment: env-a
    workstream: feature-x
    read_access:
      - namespace: env-a, workstream: main         # own env, different workstream
      - namespace: env-b, workstream: main          # cross-env, explicitly granted
    write_access:
      - namespace: env-a, workstream: feature-x     # own context only
  ```
  - Context builder opens additional namespace connections only when the grant exists
  - All cross-context reads are auditable via event log (D14)
- **What this replaces**: Earlier discussion of Docker-per-environment for SurrealDB. Docker is still used to run the single SurrealDB instance — but it's one container, not N.
- **Deferred to v2+**: Multi-organization security isolation (Company A vs Company B). Different threat model requiring either separate SurrealDB instances, separate deployments, or encryption-at-rest per tenant. Not the same problem as agent correctness isolation.
- **Alternatives considered**:
  - (A) Docker-per-environment (separate SurrealDB per env) — Rejected: solves a security problem we don't have yet. Overkill for correctness isolation. Makes cross-context access complex (network hops between containers). Resource cost scales linearly.
  - (B) Shared SurrealDB with query-level filtering only (no namespaces) — Rejected: single-layer defense. A bug in the filter logic leaks data. Namespace auth provides a second boundary.
  - (C) No isolation (single namespace, scope_id in metadata) — Rejected: no guardrails at all. Every query must include scope filters. One missing WHERE clause leaks everything.
- **Rationale**: Isolation for correctness (preventing hallucination pollution) doesn't need OS-level boundaries — it needs reliable data scoping. SurrealDB namespaces + context builder permissions provide two layers of defense. Shared infrastructure keeps resource cost constant regardless of environment count. Cross-context access (agents reading other scopes when allowed) is a natural fit for namespace-scoped connections within the same instance.

### D20: Embedding Strategy
- **Decision**: Incremental (embed on write) as default. Side-by-side column migration for model upgrades. Lazy and milestone modes available as config options.
- **Three embedding modes**:
  1. **Incremental (default)** — embed immediately on write. Knowledge is queryable the moment it's stored. Write latency adds ~5-50ms per chunk (negligible alongside Git commit + SurrealDB insert + Redb update).
  2. **Lazy (config option)** — embed on first query. Useful for bulk imports or resource-constrained environments where upfront embedding cost is unacceptable. First query per entry incurs cold-start latency.
  3. **Milestone (manual trigger)** — bulk re-embed all knowledge. Used when switching embedding models, rebuilding after corruption, or upgrading vector dimensions. Triggers the side-by-side migration mechanism.
- **Why incremental as default**:
  - Industry standard for production RAG systems (Databricks, Snowflake, Pinecone all recommend ingestion-time embedding)
  - Consistent query latency — every query hits pre-built vectors, no cold starts
  - Agent experience: no unpredictable first-query slowdowns
  - Write overhead is negligible (~5-50ms) in context of the full write pipeline
  - In an agent memory system, most knowledge WILL be queried — lazy saves compute only for knowledge that's written but never read, which should be uncommon
- **Re-embedding strategy: Side-by-side columns** (replaces PDF's blue-green namespaces):
  ```
  Phase 1: Add new column + index (non-blocking)
  ┌──────────────────────────────────────────┐
  │ knowledge_entries                         │
  │ ├── embedding_v1  [HNSW, serving reads]  │
  │ ├── embedding_v2  [HNSW CONCURRENTLY]    │  ← building in background
  │ └── content, metadata, temporal fields   │
  └──────────────────────────────────────────┘

  Phase 2: Background job re-embeds into v2
  Phase 3: Switch queries to v2 (config change)
  Phase 4: Drop v1 column + index
  ```
  - Uses SurrealDB's `DEFINE INDEX ... CONCURRENTLY` — index builds without blocking reads
  - No data duplication (non-embedding data stays in place, only embedding column is doubled temporarily)
  - Instant rollback — switch back to old column if new model underperforms
  - Progress tracking: background job records which entries are re-embedded, can pause/resume
- **Why not the PDF's blue-green namespace approach**:
  - Duplicates ALL data across namespaces, not just embeddings — wasteful
  - Conflicts with D19 (namespace-per-environment model)
  - More complex orchestration (create namespace, copy data, re-embed, swap)
  - SurrealDB's concurrent indexing makes it unnecessary
- **SurrealDB REBUILD INDEX**: Used for index maintenance (fragmentation after frequent updates), not for model migration. Rebuilds index structure but doesn't change embedding vectors. Complementary to side-by-side, not a replacement.
- **Embedding cache**: Frequently queried entries cache their vectors in memory (Redb or in-process). Reduces SurrealDB round-trips for hot knowledge. Cache invalidation on write (incremental mode handles this naturally).
- **Alternatives considered**:
  - (A) Lazy as default — Rejected: inconsistent query latency, bad agent experience. Production systems universally recommend ingestion-time embedding.
  - (B) Blue-green namespaces (PDF approach) — Rejected: designed for pgvector's limitations. SurrealDB's concurrent indexing is a simpler solution. Conflicts with namespace-per-environment (D19).
  - (C) Full re-index in place (no side-by-side) — Rejected: blocks reads during rebuild, no rollback if new model is worse.
- **Rationale**: Incremental embedding is the industry standard for good reason — consistent latency matters for agent workflows. Side-by-side column migration is the simplest zero-downtime path for model upgrades, using SurrealDB's native concurrent indexing. The PDF's blue-green was the right idea for pgvector; side-by-side columns are the right implementation for SurrealDB.

### D21: Testing Strategy *(to revisit)*
- **Decision**: Four-mode testing approach, revised from PDF's three modes to match current architecture (D15, D19).
- **Why the PDF's three modes needed revision**:
  - D19 (shared infrastructure) made the context builder the critical isolation path — needs dedicated testing the PDF didn't account for
  - D15 (trait boundary for swappability) requires contract verification — the PDF had no equivalent
  - D5 (local-first) invalidates shadow mode — no centralized production to mirror
- **Four modes**:
  1. **Unit** — Mock `QueryableStore`, `InferenceEngine`, Git ops. Tests kernel logic (merge strategies, temporal queries, namespace resolution, event log). No Docker, pure Rust `#[test]`. Sub-second.
  2. **Isolation** — Real SurrealDB with test namespaces. Dedicated tests for the correctness properties D19 depends on: agent scoped to `env-a` cannot retrieve `env-b` knowledge; agent with cross-context grant can read `env-b`; writes only go to agent's own namespace; namespace lifecycle (create/destroy). This catches the "agent reads wrong context" bug.
  3. **Integration** — Real SurrealDB + deterministic fixtures + recorded LLM responses (record/replay). Tests full pipeline: context builder → retrieval → inference → write-back. Recorded responses = deterministic + realistic + fast (no model inference at test time). Embedding pipeline uses fixed vectors for fixture data.
  4. **Contract** — Verifies each trait implementation conforms to behavioral contracts. `QueryableStore` contract: insert → search returns it, namespace isolation holds, temporal range queries correct. `InferenceEngine` contract: embed returns correct dimensions, reason returns non-empty. Critical for D15's swappability promise — when the self-built store replaces SurrealDB, contract tests validate it.
- **What replaces shadow mode**: Performance/load testing runs as a CI benchmark job with synthetic load against test namespaces. Not a "mode" — a separate concern. Doesn't need production data.
- **LLM testing via record/replay**:
  - Record: run test once with real model, capture input/output pairs
  - Replay: subsequent runs use recorded responses — deterministic, fast
  - Re-record when: model changes, prompt templates change, expected behavior changes
  - Avoids: non-determinism of live models in CI, cost of running models in every test run
- **Status**: Recorded for future revisit. Core structure is sound but details (fixture format, record/replay tooling, CI pipeline integration) need refinement during implementation.

### D22: Integration Architecture — Drop-in & Hub
- **Decision**: Three-layer integration surface: REST API + MCP Server + Rust embedded crate. REST chosen over gRPC for external-facing API.
- **Integration layers** (in priority/adoption order):
  1. **REST API** (`corvia serve`, default `localhost:8020`) — universal entry point, JSON over HTTP. OpenAI-compatible endpoints so any existing OpenAI SDK works unmodified by pointing at Corvia's base URL. Core memory endpoints:
     - `POST /v1/memories/write` — write knowledge to scope/workstream
     - `GET /v1/memories/search` — semantic search with filters
     - `GET /v1/memories/history` — temporal query (as-of, history)
     - `POST /v1/agent/session` / `DELETE /v1/agent/session` — namespace lifecycle
     - `POST /v1/embeddings` — OpenAI-compatible passthrough to InferenceEngine
  2. **MCP Server** (`corvia serve --mcp`) — agent-native protocol. Same operations exposed as MCP tools (`corvia_search`, `corvia_write`, `corvia_history`, `corvia_relate`, `corvia_promote`). Native integration for Claude, any MCP-compatible orchestrator. Already in architecture (D11) — this formalizes and expands it.
  3. **Rust crate** (`cargo add corvia`) — kernel runs in-process, zero network hop. Uniquely differentiating: mem0 and Zep are Python-only. Rust-native memory is a gap in the ecosystem.
     ```rust
     let corvia = Corvia::new(Config::local()).await?;
     let results = corvia.search("what was the API design decision?", &scope).await?;
     corvia.write("decision: use REST over gRPC", &scope).await?;
     ```
- **REST vs gRPC**: REST chosen for external API. gRPC benefits (binary efficiency, streaming) are irrelevant at memory operation latency scales. REST has universal support — every agent framework, every language, `curl`, browsers. gRPC requires protobuf codegen per language and is not browser-native. Internal kernel ↔ adapter communication uses Rust traits (D11), not a network protocol.
- **Drop-in path**: `cargo add corvia` (Rust) or point any OpenAI SDK at `http://localhost:8020` (all languages). Zero code changes for OpenAI SDK users.
- **Hub path**: `corvia serve` → everything connects to the running server via REST or MCP.
- **Python/TS SDKs**: Deferred to v2. REST covers them today — any OpenAI Python/TS library already works against Corvia's base URL.
- **Alternatives considered**:
  - (A) MCP-only — Rejected: MCP ecosystem still maturing, fewer integrations exist today vs REST. REST covers 90% of current frameworks immediately.
  - (B) gRPC instead of REST — Rejected: requires codegen in every client language, not browser-native, overkill for memory operation latency. Adoption friction outweighs performance gains that don't matter at this scale.
  - (C) REST only (no MCP) — Rejected: MCP is the native language of modern agent orchestration. Already in architecture. Not using it wastes a strategic advantage.
- **Rationale**: The Ollama playbook — OpenAI-compatible REST drove massive adoption because zero integration friction for existing users. MCP captures the agent-native market where MCP is the standard. Rust crate is the differentiator no Python-based competitor can match. Together these cover drop-in (existing stacks) and hub (Corvia-centric) usage patterns.

### Section 3: Core Kernel — APPROVED

The kernel has five subsystems, each independently testable, accessed only through Rust traits.

**1. Knowledge Store**
Manages all read/write operations against SurrealDB. Implements `QueryableStore` (D15). Handles:
- Bi-temporal entry insertion/update (D14 schema: `scope_id`, `workstream`, `valid_from`, `valid_to`, `recorded_at`, `source_version`, `superseded_by`)
- Semantic search (HNSW vector index)
- Graph traversal (`relate`, `traverse`) for cross-source dependencies (D18)
- Temporal queries (`as_of`, `history`)

**2. Agent Coordinator**
Manages agent session lifecycle in Redb (D13). Handles:
- Session open → ephemeral namespace assignment
- Heartbeat tracking + timeout/eviction
- Write buffering to agent's namespace
- On commit: enqueues writes to the merge worker
- On abort: discards namespace

**3. Embedding Pipeline**
Implements `InferenceEngine` (D5). Handles:
- Incremental embedding on write (default, D20)
- Routes to local (Ollama/vLLM) or cloud provider
- Side-by-side column migration for model upgrades (D20)
- Embedding cache (hot vectors in Redb)

**4. Context Builder**
The isolation enforcer (D19). Handles:
- Per-request permission check against agent session config
- Opens namespace-scoped SurrealDB connection(s)
- Assembles retrieval context: semantic search + graph context + temporal filter
- Ensures agents only see authorized namespaces

**5. Merge Worker**
Background process reading from Redb merge queue (D16). Handles:
- Conflict detection (two agents wrote to same entry in same scope during overlapping sessions)
- Strategy ladder: LLM merge → merge policies → last-write-wins
- Produces uniform `MergeResult` regardless of strategy
- Pre-flight conflict detection for cross-source dependencies (D18)
- Appends to `events.jsonl` audit trail (D14)

**Kernel primary trait boundaries:**
```rust
trait QueryableStore   // Storage (D15) — knowledge store, temporal/graph queries
trait InferenceEngine  // Inference (D5) — embeddings + reasoning, provider-agnostic
```

### Section 4: Adapter Layer — APPROVED

Adapters make Corvia domain-aware. The kernel is domain-agnostic — adapters bring domain knowledge to it. An adapter implements one trait and wires into the ingestion pipeline.

**The `IngestionAdapter` trait:**
```rust
trait IngestionAdapter: Send + Sync {
    fn domain(&self) -> &str;  // e.g. "git", "legal", "general"

    async fn ingest(&self, source: SourceRef, config: &AdapterConfig)
        -> impl Stream<Item = KnowledgeEntry>;

    // Optional: watch for changes and stream incremental updates
    async fn watch(&self, source: SourceRef)
        -> Option<impl Stream<Item = KnowledgeEntry>>;
}
```

An adapter chunks source material into `KnowledgeEntry` structs. The kernel handles storage, embedding, and coordination — adapters never touch SurrealDB or Redb directly.

**Git/Code adapter (first adapter, v1):**
- Source: local git repos
- Chunkers: tree-sitter AST (functions, classes, modules — carried from v0.1.0 D9), commit messages, PR/issue text
- Watch mode: `git post-commit` hook → streams incremental updates
- Output: `source_version = git_sha`, `scope_id = repo_path`

**General-purpose adapter (v2, deprioritized by D28):**
- Source: any directory of text/markdown/PDF files
- Chunkers: sliding window, paragraph-aware, heading-aware
- Watch mode: filesystem watcher (`notify` crate) for live re-ingestion
- Deprioritized from v1 to reduce scope. Git adapter is the only adapter that matters for launch.

**Third-party adapters (community, v2+):**
- Published as `corvia-adapter-*` Rust crates implementing `IngestionAdapter`
- Loaded at runtime via `Box<dyn IngestionAdapter>` — same OSS extension model as Haystack components / LlamaIndex readers, but in Rust
- Works in both embedded crate and server mode (D22)

### Section 6: Observability & Demo Visualization — APPROVED

Three consumers of observability data: operators (system health), agent developers (debugging agent behavior), demo viewers (visual showcase).

**Instrumentation: OpenTelemetry (OTLP)**

Single instrumentation API via `opentelemetry` + `tracing` Rust crates. No vendor lock-in — OTLP exports to any collector.

**What gets instrumented (mapped to kernel subsystems):**

| Subsystem | Traces | Metrics |
|-----------|--------|---------|
| Knowledge Store | `store.search`, `store.insert`, `store.traverse` | Search latency p50/p95/p99, entries per scope, graph depth |
| Agent Coordinator | `agent.session.open/close/commit/abort` | Active sessions, session duration, timeout rate |
| Embedding Pipeline | `embed.incremental`, `embed.migration` | Embed latency, throughput (chunks/sec), model version, queue depth |
| Context Builder | `context.build` (full span: permission check → retrieval → assembly) | Context size (tokens), cross-namespace reads, cache hit rate |
| Merge Worker | `merge.process`, `merge.conflict` | Queue depth, conflict rate, strategy distribution (LLM/policy/LWW), merge latency |

**Trace propagation:** Every agent request carries trace context through the full pipeline: `REST/MCP request → context builder → knowledge store queries → inference → merge worker → event log`. A single trace shows the entire lifecycle of an agent operation.

**Three-tier observability deployment:**

1. **Default (zero-config):** Local OTLP → lightweight local store. VS Code dashboard reads it. Works out of the box. No external dependencies.
2. **Demo/Dev:** `corvia serve --grafana` adds Grafana + Tempo + Prometheus containers to the Docker stack. Pre-built dashboards ship with Corvia. Off by default. Grafana is AGPL — perfect license alignment.
3. **Production (BYO):** Users export OTLP to their existing Grafana/Datadog/whatever. Corvia just emits standard OTLP.

**VS Code Dashboard (demo frontend):**

Reads from the same telemetry data — not a separate instrumentation layer. Panels:
1. **Agent Pipeline Flow** — live animated flow diagram: ingestion → embedding → storage → retrieval → inference → merge. Highlights active spans.
2. **Namespace Map** — visual representation of active scopes, workstreams, cross-namespace read grants. Shows which agents are where.
3. **Merge Activity** — queue depth over time, conflict rate, strategy breakdown.
4. **Temporal Health** — staleness heatmap per scope. Which knowledge areas are stale? Which are contested?
5. **Cost Tracker** — inference cost per scope, per agent, per time window. Shows "$0/day" prominently for local models.

**Key principle:** Dashboard is read-only and non-essential. Corvia works identically without it. The dashboard is "demo mode" — what you show people.

### D23: Observability Stack
- **Decision**: OpenTelemetry (OTLP) as single instrumentation API, three-tier deployment (local default, optional Grafana Docker stack, BYO production)
- **Grafana integration**: Optional Docker stack via `corvia serve --grafana`. Pre-built dashboards. Off by default. AGPL license alignment.
- **Alternatives considered**:
  - (A) Custom metrics/tracing — Rejected: reinventing what OTLP already does well. No ecosystem interop.
  - (B) Grafana always-on — Rejected: adds Docker containers to default setup. Violates zero-config default principle.
  - (C) Prometheus-only (no traces) — Rejected: metrics show what happened, traces show why. Agent debugging needs traces.
  - (D) VS Code dashboard only (no OTLP) — Rejected: locks observability to one frontend. Production users need their own tools.
- **Rationale**: OTLP is the industry standard. Grafana optional stack gives a powerful demo with zero effort. VS Code dashboard serves the showcase use case. Production users bring their own tools. One instrumentation layer serves all three consumers.

### Section 8: Repository Structure — APPROVED

### D24: Repository Structure
- **Decision**: Triton-inspired multi-repo under a `corvia` GitHub org. Core product in one Cargo workspace repo. Adapters, extension, and SDKs as separate repos.
- **Research**: Analyzed 9 major OSS AI infrastructure projects (Triton, vLLM, Ollama, LangChain, LlamaIndex, Haystack, mem0, MinIO, Grafana). Consistent pattern: core is 1 repo, integrations/backends are separate repos, client SDKs are separate repos per language. AGPL projects (MinIO, Grafana) license core AGPL, client SDKs Apache 2.0.
- **Triton is the closest model**: server (main executable) + core (C API) + common (shared types) + separate backend repos + separate client SDK repo. Maps cleanly to Corvia's kernel + adapters + extension + SDKs.
- **Alternatives considered**:
  - (A) Monorepo forever (Tokio/Axum model) — Rejected: couples core to complements in one repo. Adapters and extension have different release cycles, different languages. Doesn't match "core vs complement" separation (D10).
  - (B) Multi-repo from day one (every crate its own repo) — Rejected: over-fragmented. Kernel, server, CLI, and telemetry are tightly coupled — they change together and share a release cycle.
  - (C) Monorepo now, split later — Rejected: "split later" rarely happens. Architecture should match repo structure from the start.
  - (D) Two repos (core + ecosystem) — Rejected: lumps adapters and extension together. They have different languages, audiences, and release cadences.
- **Rationale**: Follows the proven pattern of every successful OSS AI infrastructure project studied. Core is one repo because kernel/server/CLI/telemetry are tightly coupled. Adapters are separate because they have independent release cycles and community can build `corvia-adapter-legal` without touching core. Extension is separate because it's TypeScript with its own toolchain. SDKs as Apache 2.0 follows MinIO/Grafana precedent — maximum adoption with SaaS protection.

### Section 10: Dogfooding Strategy — APPROVED

### D26: Dogfooding Approach
- **Decision**: Blue-green dogfooding with explicit human-controlled promotion. Corvia stable hosts and documents the development of Corvia dev. Upgrade is never automatic.
- **Alternatives considered**:
  - (A) Tag-gated — dogfood runs against latest tagged release, RC tested sequentially before tagging. Rejected: no live side-by-side comparison. Sequential testing misses issues that only appear under identical conditions.
  - (B) Canary + promotion — dev processes a subset of sources, gradually expands. Rejected: higher complexity (per-source version routing, partial rollback logic) for less benefit than blue-green. The architecture already supports full parallel runs via namespace isolation.
  - (C) No safety mechanism — just run HEAD and hope. Rejected: "sawing off the branch you're sitting on."
- **Rationale**: Blue-green is the only approach that gives true simultaneous A/B comparison on identical inputs. It dogfoods the environment isolation model (D19) — running two Corvia versions against the same SurrealDB instance IS an isolation scenario, proving the architecture works. Explicit human promotion means zero risk of accidental breakage.

**How it works:**

```
Corvia v(stable) is the HOST
  │
  ├── Running dogfood on all org repos (including corvia's own code)
  │   └── This is the knowledge base you trust and use daily
  │
  ├── You develop corvia v(dev) — commits, new features, refactors
  │   └── Corvia stable ingests these changes as knowledge
  │       (it sees your new code, indexes it, you can search it)
  │
  └── When you want to test the new version:
      │
      ├── Spin up Corvia v(dev) alongside stable
      │   └── Same sources, own SurrealDB namespace, own results
      │
      ├── Compare: does dev produce better/equal knowledge?
      │   └── Search quality, ingestion correctness, temporal queries
      │   └── Run independent test suite against dev's namespace
      │
      └── YOU choose when to promote
          └── `corvia dogfood promote` → dev becomes the new stable
          └── Old stable kept as rollback target
```

**Three phases across milestones:**

1. **Phase 1 (M1-M3, before Git adapter):** Manual dogfooding. Design docs and brainstorm files ingested via general adapter or direct REST writes. Decisions D1-D26+ become knowledge entries with bi-temporal timestamps. Validates kernel works before adapters exist.

2. **Phase 2 (M4, Git adapter lands):** Corvia stable ingests its own repos via `corvia-adapter-git`. All repos under the org are sources. Commits, code, and design docs become searchable knowledge.

3. **Phase 3 (M5, full recursion):** Blue-green is active. Stable runs dogfooding. Dev runs in parallel namespace. `.corvia/` in each repo populated by the stable pipeline. Promotion is explicit.

**Safety properties:**
- Stable is a frozen binary — new development cannot break it
- Dev only writes to its own isolated namespace (D19)
- Shared SurrealDB instance, separate namespaces — no cross-contamination
- No automatic cutover — explicit `corvia dogfood promote`
- Old stable kept as rollback target after promotion
- Bi-temporal model (D14) captures version era: `recorded_at` timestamps show which Corvia version produced each entry

**Comparison capabilities:**
- Same sources, same time window → direct A/B on search quality
- Independent test suite runs against each namespace separately
- Metrics: retrieval precision, ingestion completeness, temporal query accuracy, embedding quality
- Performance: latency, throughput, resource usage compared side-by-side

**What this proves for the portfolio:**
- Environment isolation (D19) works — demonstrated by using it on Corvia itself
- The tool can document its own evolution across version boundaries
- "Ask Corvia about Corvia" works regardless of which version produced the knowledge

## Post-Review Revisions

### D28: Developer-First Pivot
- **Decision**: Developer-first, general-capable. Revises D6 (general from day one), D25 (milestones), Section 11 (framing). Architecture unchanged — kernel stays domain-agnostic. What changes is prioritization, framing, and milestone ordering.
- **Trigger**: Strengths/weaknesses analysis revealed that "general from day one" contradicts every successful OSS adoption pattern and creates scope risk for a solo developer.
- **What changed**:
  1. **D6 revised**: Developer tooling first. Legal/clinical/financial are v2+ possibilities, not v1 promises.
  2. **Milestones reordered**: Git adapter bundled into M1 (demo-able immediately). 9 milestones → 7. General adapter deprioritized to v2.
  3. **Tagline revised**: "AI agents that understand your codebase and get smarter over time" (not "organizational reasoning memory")
  4. **General adapter deprioritized**: `corvia-adapter-general` moves from v1 bundled to v2. Git adapter is the only adapter for launch.
  5. **LinkedIn posts wait**: Published only when demo-able, not on milestone completion.
  6. **Vertical mentions cut**: No legal/clinical/financial in v1 framing.
- **What did NOT change**: Kernel architecture (stays domain-agnostic), trait system, SurrealDB, Redb, Git stores, multi-agent coordination, bi-temporal model, namespace architecture, observability, licensing, multi-repo structure. The kernel is general — the framing and prioritization are developer-focused.
- **Alternatives considered**:
  - (A) Keep general framing, just reorder milestones — Rejected: framing shapes what you build. "General" framing tempts you to build general adapters before proving the core works on one domain.
  - (B) Developer tooling only, drop general architecture — Rejected: the kernel's domain-agnostic design is good engineering regardless. It just shouldn't be the marketing pitch for v1.
- **Rationale**: Every successful OSS project started narrow: Docker ("deploy web apps"), Kubernetes ("manage Google's containers"), mem0 ("chatbot memory"), Ollama ("run Llama locally"). Build for developers first (you understand the domain, you can dogfood immediately, the demo is clear). Generalize when users ask for it.

### D29: First-Class E2E Testing & CI/CD
- **Decision**: E2E integration testing is a first-class Corvia feature, not just a developer convenience. Corvia should detect its environment, auto-provision missing dependencies (Docker, SurrealDB, vLLM), and provide a runnable test suite out of the box.
- **Scope**: Post-M1 brainstorm → design and implement.
- **What this means**:
  1. **`corvia test` subcommand**: Detects environment (what's running, what's missing), provisions what's needed, runs the full test suite, tears down.
  2. **Environment detection**: Is Docker available? Is SurrealDB reachable? Is vLLM/Ollama serving embeddings? Report status and auto-fix where possible.
  3. **CI/CD dual-mode**: Runnable locally as a pre-push check AND as a GitHub Action. Same test suite, same assertions, different infrastructure setup.
  4. **Dogfooding alignment (D26)**: The test infrastructure itself demonstrates Corvia's value — an AI-agent-first repo that can bootstrap and validate its own stack.
- **Why not in M1**: M1's goal is "point at repo, search code" — the demo loop. E2E test infrastructure is important but doesn't block the core demo. Designing it properly (env detection, auto-setup, CI/CD config) deserves its own brainstorm.
- **Rationale**: If Corvia claims to be the organizational memory for AI agents, its own test infrastructure should demonstrate that claim. A tool that can bootstrap its own dependencies for testing is a powerful portfolio and OSS signal.

### D30: Recursive Self-Validation (Introspect)
- **Decision**: Corvia's E2E testing is not just infrastructure verification — it is recursive self-validation. Corvia ingests its own source code, queries its own knowledge, and validates that it can understand itself. Named `Introspect`.
- **Alternatives considered**:
  - (A) Conventional integration test (hardcoded test strings, check mechanics) — Rejected: tests pipes, not value. Doesn't catch chunking/embedding/search quality regressions.
  - (B) `corvia doctor` + `corvia test` (two commands, separate diagnosis from testing) — Rejected: clean separation but more surface area. `--check-only` flag achieves the same.
  - (C) `corvia init --verify` (extend existing init) — Rejected: conflates initialization with testing. Can't re-test without re-init.
- **Rationale**: Claude Code principle — "the tool uses itself to build itself." 90% of Claude Code is written by Claude Code. For Corvia, the equivalent: if Corvia can't understand its own codebase, it can't understand anyone else's. Self-ingestion exercises the full pipeline with real code, not synthetic strings.

### D31: Dual-Purpose Pipeline (Test + Demo)
- **Decision**: `corvia test` and `corvia demo` share the same Introspect pipeline (env check, self-ingest). `test` runs canonical queries with assertions and exit codes. `demo` drops into an interactive REPL where users search Corvia's own code.
- **Rationale**: One pipeline, two audiences. Developers get recursive validation for CI/CD. New users get the most honest demo possible — Corvia searching its own codebase. The demo IS the test; the test IS the demo.

### D32: Canonical Queries in TOML
- **Decision**: Self-validation queries and expected results are stored in `tests/introspect.toml`, not hardcoded. Each query has text, expected top-result file, and minimum score threshold.

## Hybrid Embedded-First Architecture (D35-D42)

**Context**: After completing M1 with SurrealDB as the queryable store, a competitive analysis and architectural challenge session (2026-03-01) questioned whether the three-store architecture (Git + SurrealDB + Redb) was justified from day one. Key finding: every feature except advanced graph SQL (centrality, community detection) can be implemented with embedded Rust components — zero Docker required. This led to a two-tier storage design that preserves the `QueryableStore` trait boundary while offering a dramatically simpler default experience.

**Competitive landscape at time of decision**: Letta (MemGPT) shipped Context Repositories (git-backed agent memory with worktrees), Mem0 raised $24M for conversational memory, Cognee raised $7.5M for hybrid vector+graph. GitHub Copilot Memory entered public preview. No competitor combines git-as-truth + vector search + graph traversal + multi-agent coordination without Docker. The "good enough" built-in memory solutions (Copilot Memory, CLAUDE.md, Windsurf Memories) are the primary adoption threat — zero-friction onboarding is critical.

### D35: Two-Tier Storage Architecture — LiteStore (default) + FullStore (opt-in)
- **Decision**: Corvia ships with two storage backends behind the same `QueryableStore` trait. LiteStore is the default (zero Docker, full-featured). FullStore is an opt-in upgrade for advanced graph SQL and scale.
- **LiteStore components**: hnswlib-rs (HNSW vector search), petgraph (graph traversal), Redb (metadata, coordination, temporal index, edge storage), Ollama (embeddings), Git (source of truth)
- **FullStore components**: SurrealDB (replaces hnswlib-rs + Redb query layer), vLLM (replaces Ollama for throughput), Git (source of truth), Redb (coordination only)
- **CLI**: `corvia init` → LiteStore. `corvia init --full` → FullStore. `corvia upgrade` → migrate LiteStore to FullStore.
- **What LiteStore covers**: Vector search, graph traversal (BFS/DFS/paths/cycles), multi-agent coordination, LLM-assisted merge, bi-temporal queries, application-level RBAC isolation, full namespace system
- **What FullStore adds**: SurrealDB graph SQL (centrality, community detection, PageRank), multi-process concurrent access, distributed deployment, ad-hoc SurrealQL
- **Resource comparison**: LiteStore: ~50-150MB RAM, <1s startup, zero Docker. FullStore: ~200-500MB RAM, ~30s first run, Docker required.
- **Alternatives considered**:
  - (A) LiteStore as basic search only, FullStore for everything else — Rejected: creates an artificially limited free tier. Most features work without Docker.
  - (B) SurrealDB embedded mode — Rejected (D15): BSL 1.1 ↔ AGPL v3 license conflict via static linking.
  - (C) SQLite-vec for vectors + metadata — Rejected: adds C FFI dependency to pure-Rust project. hnswlib-rs + Redb covers the same ground natively.
- **Rationale**: The `QueryableStore` trait boundary (D15) makes this a natural evolution, not a rewrite. Every feature that can work embedded should work embedded. Docker is justified only when the capability genuinely requires a separate process. Zero-Docker onboarding directly addresses the "good enough" adoption threat from built-in IDE memory solutions.

### D36: LiteStore Vector Search — hnswlib-rs
- **Decision**: Use `hnswlib-rs` (pure Rust HNSW implementation) for in-process vector search. Index persisted to `.corvia/index.hnsw` via built-in dump/reload. Memory-mapped for fast startup.
- **Why hnswlib-rs over alternatives**:
  - `hnswlib-rs`: Pure Rust, concurrent search + concurrent mutation, dump/reload to disk, actively maintained. Apache 2.0 license.
  - `hnsw_rs`: Similar, but less documentation and smaller community.
  - `USearch`: C++ with Rust bindings (FFI), faster but adds C dependency.
- **Index lifecycle**: Built incrementally on insert. Persisted to disk periodically and on shutdown. Loaded via mmap on startup (<100ms). Full rebuild from Git-tracked knowledge files via `corvia rebuild`.
- **Scope filtering**: Post-filter after KNN retrieval (retrieve K×2 candidates, filter by scope, return top K). Acceptable for <100K entries per scope. For larger scales, FullStore is recommended.
- **Performance**: ~10ms search at 10K entries, ~20ms at 100K. Single-digit millisecond at <5K.

### D37: Graph Overlay — petgraph + Redb Edge Storage
- **Decision**: Use `petgraph` (Rust graph library, used by cargo itself) for in-memory graph traversal. Edges persisted in Redb table. Graph rebuilt from Redb on startup. Supports BFS, DFS, shortest path, topological sort, cycle detection.
- **Edge storage**: Redb table `graph_edges: (source_id, edge_type, target_id) → metadata`. Loaded into petgraph `DiGraph` on startup.
- **Trait extension**: Add `relate()`, `edges()`, `traverse()`, `shortest_path()` to `QueryableStore` trait. LiteStore implements via petgraph. FullStore implements via SurrealDB `RELATE`/graph queries.
- **FullStore extras**: `centrality()`, `community_detect()` — available only in FullStore. These methods return `Err(NotSupported)` in LiteStore, or live on a separate optional `AdvancedGraphStore` trait.
- **Alternatives considered**:
  - (A) Store edges in Git JSON only, rebuild petgraph from files — Rejected: slower startup, Redb is already available and faster.
  - (B) Skip graph in LiteStore entirely — Rejected (this session): multi-hop traversal is doable and valuable without Docker.
- **Rationale**: petgraph is the standard Rust graph library (1,800+ stars). In-memory graph is fast for the scale LiteStore targets (<50K nodes). Redb persistence means no rebuild from Git on every startup.

### D38: Bi-Temporal Overlay — Redb Range Index
- **Decision**: Bi-temporal queries implemented via Redb compound-key range scans. No SurrealDB required.
- **Redb table**: `temporal_index: (scope_id, valid_from_millis, entry_id) → (valid_to_millis, recorded_at_millis)`. Ordered by compound key, enabling efficient range scans.
- **Query: `as_of(scope, T)`**: Range scan where `valid_from <= T`, filter where `valid_to > T` or `valid_to IS NULL`. Returns all entries valid at time T.
- **Query: `history(entry_id)`**: Follow the `superseded_by` chain through Redb lookups. Returns all versions of an entry.
- **Trait extension**: Add `as_of()` and `history()` to `QueryableStore`. Both backends (LiteStore and FullStore) implement.
- **Git tracking**: Knowledge JSON files already contain `valid_from`, `valid_to`, `recorded_at`, `superseded_by` fields (types.rs:15-18). Redb index is rebuilt from these on `corvia rebuild`.
- **Rationale**: The data is already in `KnowledgeEntry`. The only missing piece was a way to query it efficiently. Redb's B-tree ordered keys give O(log n) range scans — sufficient for temporal queries at LiteStore scale.

### D39: Application-Level RBAC Isolation
- **Decision**: Correctness isolation (D19) enforced at application level in LiteStore, without SurrealDB namespace auth. Three layers of defense.
- **Layer 1 — Context Builder**: Checks agent session config before every read/write. Agent declares which namespaces it can access at session start. All queries pass through Context Builder.
- **Layer 2 — Redb key prefixes**: All Redb keys prefixed with `{namespace}:{scope}:`. A scan for scope-A data cannot accidentally return scope-B data because the key prefix differs.
- **Layer 3 — File directory structure**: `.corvia/knowledge/{namespace}/{scope}/` — filesystem boundaries reinforce logical boundaries.
- **What this does NOT provide**: Security isolation against adversarial access. A malicious process with disk access can read any file. This is correctness isolation (preventing accidental cross-scope pollution), same threat model as D19.
- **FullStore adds**: SurrealDB namespace-level auth (DB connections scoped to namespace with RBAC credentials). Defense-in-depth for production multi-user scenarios.
- **Rationale**: D19 itself states isolation is for correctness, not security. Application-level enforcement with key prefixes and directory structure provides the same correctness guarantees as SurrealDB namespaces for single-machine, single-user scenarios.

### D40: Embedding Engine — Ollama Default, vLLM Opt-In
- **Decision**: `OllamaEngine` is the default `InferenceEngine` implementation. `VllmEngine` (current) becomes opt-in for FullStore / power users. Both implement the same `InferenceEngine` trait (D5).
- **Ollama API**: `POST http://localhost:11434/api/embed` (single) or OpenAI-compatible `POST http://localhost:11434/v1/embeddings` (batch).
- **Default model**: `nomic-embed-text-v1.5` (274MB, 768 dimensions) — same model as current vLLM setup. Consistency across engines.
- **Auto-setup**: `corvia init` checks if Ollama is installed and model is pulled. Provides install instructions if missing. Does NOT auto-install Ollama — respects that Ollama is a system-level tool.
- **Performance on small systems**: CPU-only laptop: ~30-60 embed/sec. Initial ingest of 10K chunks: ~3-5 minutes. Search latency: ~20-50ms per query (single embed call). Ingest is a one-time cost; search is fast regardless.
- **Config**: `corvia.toml` specifies `inference.provider = "ollama"` (default) or `"vllm"`. URL, model, and dimensions configurable.
- **Alternatives considered**:
  - (A) Keep vLLM only — Rejected: requires Docker, contradicts zero-Docker LiteStore goal.
  - (B) Auto-detect (try Ollama, fall back to vLLM) — Rejected: implicit behavior confuses debugging. Explicit config is better.
  - (C) Embedded Rust model (candle/burn) — Rejected: adds significant binary size (~500MB+), GGUF model loading complexity, and GPU abstraction. Ollama already solves this well.
- **Rationale**: Ollama is to LLM inference what Docker is to containerization — a simple wrapper that handles the complexity. It fits the "one native install + corvia" onboarding story. No Docker, no Python, no CUDA setup.

### D41: Knowledge File Format — Git-Tracked JSON
- **Decision**: Knowledge entries stored as individual JSON files in `.corvia/knowledge/{namespace}/{scope}/`. Git-tracked. Embeddings stored in the same JSON file.
- **File path**: `.corvia/knowledge/{namespace}/{scope}/{entry_id}.json`
- **File content**: Full `KnowledgeEntry` serialized as JSON, including embedding vector.
- **Why include embeddings in JSON**: Enables `corvia rebuild` without re-embedding. Re-embedding 10K entries takes minutes on CPU. Storing vectors saves rebuild time.
- **Why not sidecar files**: Simpler — one file per entry, atomic write, atomic delete. Sidecar `.vec` files add coordination complexity.
- **Git implications**: Embedding vectors as JSON arrays are ~3-6KB per 768-dim vector. A repo with 10K entries: ~30-60MB of JSON. Acceptable for git, but `.corvia/knowledge/` can be gitignored if size is a concern (recovery via `corvia rebuild` from source).
- **Gitignore strategy**: `.corvia/index.hnsw` and `.corvia/coordination.redb` are always gitignored (ephemeral). `.corvia/knowledge/` is git-tracked by default. User can gitignore it and rely on `corvia rebuild` from source code.
- **Alternatives considered**:
  - (A) Binary format (MessagePack/CBOR) — Rejected: not human-readable, defeats transparency benefit.
  - (B) Embeddings in separate binary file — Rejected: coordination complexity for marginal space savings.
  - (C) No file storage, Redb only — Rejected: Redb is not git-trackable. Loses "source of truth in git" principle.
- **Rationale**: JSON files in git give Letta's transparency (agents can browse, humans can inspect) with Corvia's structure (typed entries, embeddings included, rebuildable indexes). Best of both worlds.

## Agent Coordination & Milestone Revision (D43-D45)

*Added: 2026-03-01 — Research-driven decisions from competitive analysis, orchestrator landscape survey, and agent identity standards review.*

### D43: Agent Write Isolation — Staging Hybrid (Git Branches + Shared HNSW)

- **Decision**: Agent write isolation uses lightweight staging directories + git branches, NOT full git worktrees or Redb-only logical isolation. The HNSW index and Redb metadata remain shared (single instance). Git branches provide audit trail.
- **Alternatives considered**: (1) Full git worktrees — rejected because HNSW binary index can't be forked per worktree without full duplication (~50-100MB per agent), knowledge files would duplicate (~10K × 5KB per agent), and cross-agent search requires shared index. (2) Redb-only logical isolation — rejected because no native git history per agent, contributions invisible in `git log`, rollback requires scanning Redb key prefixes vs simple `git branch -D`.
- **Write path**: Agent starts session → gets staging dir `.corvia/staging/{agent-id}/` + git branch `{agent-id}/session-{uuid}` → writes knowledge (JSON to staging, vector into shared HNSW tagged `{agent-id}:pending`) → commits on agent branch → merge worker processes (auto-merge or LLM-assisted merge) → post-merge moves JSON to `.corvia/knowledge/`, retags Redb metadata, merges git branch into main, cleans up.
- **Read path**: Agent searches shared HNSW → Redb metadata filter (scope, workstream, permissions) → results include main knowledge + agent's own pending. Visibility configurable: `own`, `all`, `explicit`.
- **Embedding for conflicts**: Non-conflicting writes keep embedding as-is. Conflicting writes: merge worker produces merged content → re-embeds → replaces vectors.
- **Disk overhead**: ~10KB per agent session (staging only), not 50-100MB (worktrees).
- **Rationale**: Gives git audit trail (`git log --all`, `git blame`) at minimal cost. Shared HNSW means cross-agent search is instant. Letta uses worktrees because memory IS files (markdown); Corvia's knowledge is structured data with embeddings where the index is the primary query path.

### D44: Agent Capability Scope — Levels 0-3 (Memory + Reasoning + Findings)

- **Decision**: Corvia targets capability Levels 0-3. Code writing (Level 4) is explicitly out of scope — separate project or open-source integration.
- **Levels**: Level 0 (Passive Index: ingest, embed, search — M1 done). Level 1 (Knowledge Writer: agents create/update entries, merge conflicts — M2). Level 2 (Knowledge Reasoner: cross-reference, detect gaps, surface contradictions — M3). Level 3 (Structured Findings: detect patterns, post findings as knowledge entries, no diff generation — M3).
- **Level 3 revised scope**: Originally "generate diffs stored as knowledge entries." Revised to "detect patterns and post structured findings" after orchestrator landscape analysis revealed this would overlap with orchestrator territory. Corvia is a "suggestion board" — posts findings, external coding agents pick them up via MCP, generate their own diffs. Corvia tracks acceptance rate.
- **Why not Level 4**: Code writing agents already exist (Claude Code, Cursor, Copilot, Devin). Corvia's value is *informing* those agents, not replacing them. Keeps trust model simple — Corvia never modifies the codebase it indexes.
- **Expanded git adapter vision**: Beyond source files to also ingest git commits, diffs, commit messages. Future `corvia-adapter-github`: PRs, issues, CI results, review comments.
- **Research basis**: Surveyed OpenHands (65k stars), AutoGen (50k), MetaGPT (46k), CrewAI (44k), LangGraph (25k), SWE-agent (10.5k), Rust frameworks (Rig, OpenFang). No system produces structured findings as first-class artifacts flowing between agents via a knowledge layer — this is Corvia's niche.

### D45: Agent Identity, Session Recovery & Observability Contract

- **Decision**: Multi-layer agent identity for ecosystem compatibility. Agents have persistent human-readable IDs internally. External agents from any framework connect via MCP with optional identity metadata. Sessions are ephemeral work units that survive crashes. Observability is designed in at lifecycle definition time.
- **Eight-part design**:

**Part 1 — Multi-Layer Identity**:
- `Registered`: Internal Corvia agent. Format: `{scope}::{agent-name}` (e.g., `myproject::code-indexer`). Full staging hybrid access.
- `McpClient`: External agent via MCP. Identified by `clientInfo.name` + optional `_meta.agent_id` in tool calls. Write access if `_meta.agent_id` present, read-only otherwise.
- `ActorToken`: Future IETF `actor_token` standard. JWT `sub` claim as agent identity.
- `Anonymous`: No identity. Read-only, rate-limited.
- **Industry context**: No standard for agent-level identity exists (2026-03-01). MCP identifies applications, not agents. IETF `actor_token` and W3C `did:wba` are 1-2 years out. MCP's `_meta` field is the pragmatic bridge today — zero spec changes needed.

**Part 2 — Internal Identity Scheme**:
- Agent ID: `{scope}::{agent-name}`. Session ID: `{agent-id}/sess-{uuid}`. `::` separator distinguishes from 5-segment namespace (`:` delimited).
- Redb `agents` table: `agent_id`, `display_name`, `identity_type`, `registered_at`, `permissions` (ReadOnly/ReadWrite/Admin), `last_seen`, `status` (Active/Suspended/Deregistered).

**Part 3 — Knowledge Ownership**:
- `KnowledgeEntry` extended with: `agent_id: Option<String>`, `session_id: Option<String>`, `status: EntryStatus` (Pending → Committed → Merged or Rejected).
- For McpClient agents: `agent_id` = `"{client_name}::{agent_hint}"` if `_meta` present.

**Part 4 — Atomic Write Path** (revision of D43 write path):
- Step 1: Write JSON to staging dir (filesystem — cheap, recoverable). Step 2: Embed via Ollama/vLLM (idempotent — same content = same vector). Step 3: Redb transaction (ATOMIC — entry metadata + HNSW mapping + vector insert, all-or-nothing).
- No ghost vectors possible. Recovery scan on resume: for each staging file without matching Redb entry → re-embed → atomic insert.

**Part 5 — Session Lifecycle & Crash Recovery** (Registered agents only):
- State machine: Created → Active (heartbeat) → Committing → Merging → Closed. Active → Stale (5min timeout) → Orphaned (20min grace) → Recoverable.
- Reconnection: Agent presents `agent_id` → Corvia returns recoverable sessions → agent chooses resume/commit/rollback/ignore. Agent remembers nothing; Corvia remembers everything.
- Commit flow: 5 idempotent steps (session status → git commit → entry status → merge queue → session status). Crash at any step → replay from beginning, completed steps are no-ops.
- Merge worker: conflict detection (semantic similarity) + LLM merge. Crashes → replays queue. LLM failure → exponential retry (max 3) → stays in queue for human intervention.
- McpClient agents: no crash recovery. Writes go direct to merge queue. In-flight MCP calls lost on disconnect (per MCP spec). Completed writes safe.

**Part 6 — Garbage Collection**:
- Orphaned sessions > 24h → auto-rollback. Closed sessions > 7d → delete Redb record. Inactive agents > 30d → Suspended.
- All thresholds configurable in `corvia.toml` under `[agent_lifecycle]`.

**Part 7 — API Surface**:
- REST (registered): `POST /agents`, `POST /agents/{id}/sessions`, `POST /sessions/{id}/heartbeat|commit|rollback|recover`, `GET /agents/{id}/knowledge`, `GET /sessions/{id}/state`.
- MCP (all): `corvia_search` (all), `corvia_write` (Registered + McpClient with `_meta`), `corvia_history` (all), `corvia_agent_status` (Registered + McpClient with `_meta`).

**Part 8 — Observability Contract**:
- OpenTelemetry spans: `corvia.agent.register`, `corvia.session.create`, `corvia.entry.write` (with `corvia.entry.embed` + `corvia.entry.insert` sub-spans), `corvia.session.commit`, `corvia.merge.process` (with `corvia.merge.conflict` + `corvia.merge.llm_resolve` sub-spans), `corvia.gc.run`, `corvia.search`.
- Metrics (gauges): `agents.active`, `sessions.active`, `sessions.stale`, `entries.pending`, `merge.queue_depth`. (Counters): `entries.merged`, `entries.rejected`, `gc.sessions_cleaned`, `merge.conflicts`. (Histograms): `embed.duration_ms`, `search.duration_ms`, `merge.llm_duration_ms`, `heartbeat.delta_ms`.
- Structured log events: `agent_registered`, `session_created`, `session_stale` (WARN), `session_recovered`, `entry_merged`, `merge_failed` (WARN), `gc_sweep`, `mcp_client_connected`, `mcp_write_denied` (WARN).
- M2/M3 code emits telemetry from day one. M4 configures export targets (stdout/file for LiteStore, OTLP/Grafana for FullStore).

## Workspace Model & Devcontainer Demo (D46-D52)

*Added: 2026-03-01 — Designing how Corvia operates across multiple repos, and a devcontainer-based demo that showcases it.*

### D46: Workspace Model — Purpose-Built Directory

- **Decision**: A Corvia workspace is a purpose-built directory that contains repo clones/symlinks and a `.corvia/` knowledge store. Target repos are NOT polluted with Corvia artifacts. The workspace is the unit of multi-repo organization.
- **Alternatives considered**: (1) Corvia inside target repo (`.corvia/` in target repo, gitignored) — rejected because it pollutes the target repo's directory structure, awkward for multi-repo. (2) Central registry (`~/.corvia/projects/`) — rejected because not portable, tied to machine filesystem paths, devcontainer/demo needs path reconstruction.
- **Directory structure**:
  ```
  my-workspace/
  ├── corvia.toml          ← has [workspace] section
  ├── .corvia/             ← knowledge store, indexes, coordination
  │   ├── .git/            ← knowledge git (source of truth)
  │   ├── knowledge/       ← git-tracked JSON knowledge files
  │   ├── hnsw_index/      ← LiteStore vector index
  │   └── coordination.redb
  ├── repos/               ← cloned or symlinked target repos
  │   ├── repo-a/
  │   └── repo-b/
  └── .devcontainer/       ← optional
  ```
- **Rationale**: Same pattern as Cargo workspaces, Nx workspaces, Docker Compose projects. Orchestrates multiple concerns without polluting any individual repo. Portable — move the directory, everything moves with it.

### D47: Workspace Config — Unified `corvia.toml` with `[workspace]` Section

- **Decision**: Workspace configuration is a `[workspace]` section inside `corvia.toml`, not a separate file. If `[workspace]` is present, the directory is a workspace. If absent, it's a single-project (backward compatible with M1/M2).
- **Alternatives considered**: Separate `corvia-workspace.toml` — rejected because it splits concerns across two files unnecessarily. Cargo uses one `Cargo.toml` with optional `[workspace]`; same pattern works here.
- **Config format**:
  ```toml
  [project]
  name = "my-project"
  scope_id = "my-org"

  [workspace]
  repos_dir = "repos"

  [[workspace.repos]]
  name = "backend"
  url = "https://github.com/org/backend"
  local = "/home/dev/backend"           # optional local override
  namespace = "backend"

  [[workspace.repos]]
  name = "frontend"
  url = "https://github.com/org/frontend"
  namespace = "frontend"
  ```
- **Namespace mapping**: Each repo's `namespace` field becomes the `workstream` segment in the 5-segment namespace `{org}:{scope_id}:{workstream}:{source}:{version_ref}`.
- **Local override behavior**: If `local` path exists and is a valid git repo, use it (no clone). Otherwise fall back to cloning from `url` into `{repos_dir}/{name}`.

### D48: Workspace CLI — First-Class Subcommand Group

- **Decision**: `corvia workspace` is a first-class subcommand group, not a directory convention or thin wrapper.
- **Alternatives considered**: (1) Directory convention only (no new commands, `corvia init` detects workspace.toml) — rejected because undiscoverable, nothing in `--help` tells you workspaces exist. (2) Thin wrapper (only `corvia workspace init`, then existing commands) — rejected because workspaces are a core concept deserving full CLI support.
- **Commands**:
  - `corvia workspace create <name> --repo <url> [--template <url|path>] [--local <name>=<path>]` — scaffold + init + first ingest
  - `corvia workspace init` — initialize existing directory (clone missing repos, set up `.corvia/`)
  - `corvia workspace status` — show repos, ingest state, knowledge counts per namespace
  - `corvia workspace add <url> [--name] [--namespace] [--local]` — add a repo
  - `corvia workspace remove <name> [--purge]` — remove a repo (optionally delete cloned repo + knowledge)
  - `corvia workspace ingest [--repo <name>] [--fresh]` — ingest all or one repo
- **Existing command adaptation**: `corvia ingest` (no path) in a workspace behaves as `corvia workspace ingest`. `corvia search` shows namespace (repo origin) in results. `corvia status` shows workspace info.

### D49: Repo Sourcing — Declare URLs, Allow Local Overrides

- **Decision**: Workspace config declares canonical repo URLs for reproducibility. Developers can set `local` path overrides to use existing checkouts instead of cloning.
- **Rationale**: URLs make the workspace portable and reproducible (anyone can `corvia workspace init` and get the same repos). Local overrides avoid re-cloning repos a developer already has, and support push access to existing checkouts.

### D50: Demo/Template Repo — Separate `corvia-workspace` Repository

- **Decision**: A separate `corvia-workspace` repo serves three roles: (1) **Template** — `corvia workspace create --template corvia-workspace` scaffolds new workspaces. (2) **Demo** — clone + open in Codespaces/DevPod for instant 1-click Corvia experience. (3) **Development environment** — the workspace we use to develop Corvia M3+ onward.
- **Alternatives considered**: (1) Inside corvia monorepo (`demo/` directory) — rejected because bloats main repo with pre-ingested knowledge, "try it" URL points at subdirectory not standalone repo. (2) Inside adapter repo — rejected because self-referential (demo indexes the repo it lives in), adapter is a library crate that users wouldn't look at for demos.
- **Triple-role principle**: If the workspace is broken or awkward, we feel it immediately because we develop inside it. Knowledge accumulates across milestones. The demo stays current because we use it daily.
- **Demo content**: Corvia (main kernel) + corvia-adapter-git (the integration point). Authentic multi-repo with real cross-project relationships (e.g., `IngestionAdapter` trait in kernel ↔ `GitAdapter` impl in adapter).

### D51: Demo Startup — Pre-Ingested Per Release, `--fresh` Flag

- **Decision**: The demo repo ships with pre-ingested `.corvia/knowledge/` and `hnsw_index/`. Users get instant search on first open. A `--fresh` flag (`corvia workspace ingest --fresh`) lets users rebuild from scratch.
- **Release process**: Each Corvia release regenerates the pre-ingested knowledge. CI step: checkout both repos at release tag → `corvia workspace ingest` → commit updated `.corvia/knowledge/` to `corvia-workspace` → tag release.
- **Alternatives considered**: (1) Ingest on first open — rejected because 2-5 minute wait on first startup is poor UX for a demo. (2) Pre-ingested seed + incremental — unnecessary complexity when full pre-ingestion is feasible.

### D52: Devcontainer as Integration Surface

- **Decision**: The demo uses a devcontainer for delivery — an open standard (not VS Code-specific). The devcontainer provides the multi-repo workspace environment, NOT Corvia's infrastructure. Corvia remains zero-Docker (D35 LiteStore). Ollama runs as a process inside the container alongside Corvia.
- **Key insight**: Corvia itself is zero-Docker (LiteStore). The devcontainer wraps the *environment* (workspace + repos + Ollama + Corvia binary), not Corvia's storage or processing. This preserves D35 while enabling reproducible 1-click demos.
- **IDE-agnostic**: Devcontainer spec is open standard. Works with VS Code, Cursor, GitHub Codespaces, JetBrains Gateway, DevPod, devcontainer-cli. Aligns with D22 (REST + MCP + Rust crate integration surface — not tied to one IDE).
- **Lifecycle hooks**: `postCreateCommand` runs `corvia workspace init` (clones repos, pulls Ollama model). `postStartCommand` runs `corvia serve --mcp &` (background server). Pre-ingested knowledge means instant search without waiting for ingest.
- **Milestone placement**: M2.1 — focused sub-milestone after M2 (agent coordination complete), before M3 (temporal + graph). Current M2 capabilities (ingest, search, agent coordination) are sufficient to demo.
- **Implementation approach**: Parallel tracks (Approach C). Define config format first (the keystone contract), then alternate between workspace CLI (Stream 1) and demo repo setup (Stream 2).

## Introspect Design (D29-D33)

### Design Principles

**Recursive Development (Claude Code Principle):**
Claude Code is 90% written by Claude Code. The tool uses itself to build itself, creating a self-reinforcing quality loop. For Corvia, the equivalent: **Corvia ingests its own code, queries its own knowledge, and validates that it understands itself.** If Corvia can't understand its own codebase, it can't understand anyone else's.

**Dual-Purpose Pipeline:**

| Developer (recursive dev) | New user (demo) |
|---------------------------|-----------------|
| Validates changes | Sees Corvia work |
| Catches regressions | Tries real queries |
| CI/CD quality gate | Understands the value prop |
| "Did I break search?" | "Oh, THIS is what code memory does" |

**Minimal New Code:**
`Introspect` is an orchestrator, not a new system. It composes: `DockerProvisioner` (env check + auto-provisioning), `GitAdapter` (self-ingestion via tree-sitter), `VllmEngine` (embedding), `SurrealStore` (storage + search).

### Component Map

```
corvia-kernel/
  src/
    introspect.rs        <- NEW: Introspect pipeline struct
    docker.rs            <- EXTEND: add vLLM container provisioning

corvia-cli/
  src/
    main.rs              <- EXTEND: add `test` and `demo` subcommands

corvia-common/
  src/
    config.rs            <- EXTEND: env var overrides (CORVIA_*)

tests/
  introspect.toml        <- NEW: canonical queries + expected results

.github/
  workflows/
    introspect.yml       <- NEW: GitHub Action for CI
```

### Data Flow

```
corvia test / corvia demo
  |
  +-- Phase 1: check_env()
  |     DockerProvisioner::is_running() for SurrealDB + vLLM
  |     Auto-provision if missing
  |
  +-- Phase 2: ingest_self()
  |     GitAdapter::ingest(".") on Corvia's own repo
  |     VllmEngine::embed_batch() on all chunks
  |     SurrealStore::insert() into corvia_introspect namespace
  |
  +-- Phase 3: query_self()          [test mode only]
  |     Load canonical queries from tests/introspect.toml
  |     VllmEngine::embed() each query
  |     SurrealStore::search() and check against expected files
  |
  +-- Phase 3 alt: repl_loop()       [demo mode only]
  |     Read query from stdin
  |     Embed + search + display results
  |     Loop until exit/Ctrl+C
  |
  +-- Phase 4: report() + teardown
        Pass/fail per query, aggregate score
        Delete corvia_introspect namespace (unless --keep)
        Exit code: 0 (pass) / 1 (quality fail) / 2 (infra fail)
```

### Canonical Queries

Stored in `tests/introspect.toml`, evolve with the codebase:

```toml
[config]
default_min_score = 0.70
scope_id = "corvia-introspect"

[[query]]
text = "how does embedding work?"
expect_file = "crates/corvia-kernel/src/embedding_pipeline.rs"
min_score = 0.70

[[query]]
text = "how is knowledge stored in the database?"
expect_file = "crates/corvia-kernel/src/knowledge_store.rs"
min_score = 0.70

[[query]]
text = "what CLI commands are available?"
expect_file = "crates/corvia-cli/src/main.rs"
min_score = 0.65

[[query]]
text = "how does tree-sitter chunk code?"
expect_file = "src/treesitter.rs"
min_score = 0.70
```

Why TOML, not hardcoded: queries evolve with the codebase, contributors add queries when they add features, `corvia test --baseline` regenerates scores, CI diffs baselines to catch regressions. Hardcoded fallback compiled into binary if TOML is missing.

### CLI Surface

```
corvia test                   # full introspect pipeline (4 phases)
corvia test --check-only      # phase 1 only (env diagnosis)
corvia test --keep            # skip teardown (for debugging)
corvia test --ci              # strict thresholds + JSON output
corvia test --baseline        # record current scores as new baseline

corvia demo                   # phases 1+2, then interactive REPL
corvia demo --keep            # preserve data after exit
```

All introspect data goes into a dedicated SurrealDB namespace (`corvia_introspect`). This never pollutes real user data. Teardown deletes the entire namespace.

### Error Handling

**Exit Codes (POSIX standard):**

| Code | Meaning | Example |
|------|---------|---------|
| `0` | Success | All introspect phases passed |
| `1` | Test failure | Query quality regression |
| `2` | Usage/infra error | Docker unavailable, bad config |

**Phase 1 (check_env):** Docker missing → exit 2 with install instructions. SurrealDB down → auto-provision via DockerProvisioner. vLLM down → auto-provision via DockerProvisioner (new vLLM support). vLLM no GPU → exit 2 with guidance.

**Phase 2 (ingest_self):** Repo not found → fallback to cwd. No parseable files → exit 2. Embedding API error → retry once, then exit 2.

**Phase 3 (query_self):** introspect.toml missing → use built-in default queries. Scores below threshold → report FAIL, continue remaining. Zero results → report FAIL with diagnostic info.

### CI/CD Integration

**GitHub Action:**


### Goose Comparison (Context)

Block's Goose headless mode was analyzed during this design. Key differences: Goose is an AI agent in CI (LLM reasoning on tasks); Corvia Introspect is infrastructure self-test (deterministic assertions on its own knowledge). Goose assumes dependencies exist; Corvia auto-provisions them. Goose is non-deterministic (LLM output); Corvia Introspect is deterministic (embed + search + score threshold).

## Python Components (D53-D55)

*Session: 2026-03-01 (Python use-case brainstorm — where Python is genuinely superior to Rust)*

**Context:** Corvia is a Rust project by design — async/concurrency without GIL, trait-based swappability, and performance justify this. However, three components exist where Python has a **genuine technical advantage** over Rust, not merely ecosystem preference. Each maps to an existing milestone where it's the natural implementation language.

### D53: Python SDK with Rust Backend via PyO3 (`corvia-py`) — M5/M7

**Decision:** Ship a PyO3-backed Python package (`pip install corvia`) that embeds the full Rust kernel — LiteStore, OllamaEngine, GitAdapter — directly callable from Python without a running server.

**Why this approach:** The consumers are Python (LangChain, CrewAI, LlamaIndex, AutoGen, Jupyter). But rather than a thin HTTP wrapper, PyO3 gives Python users the **full kernel embedded** — like how `polars` embeds a Rust DataFrame engine or `duckdb` embeds a C++ query engine. `pip install corvia` and you have a working knowledge store. No `corvia serve` needed for basic operations.

**Architecture:**

```
Python user code
  ├─→ PyO3 bindings (corvia-python crate)  → Rust kernel directly (embedded mode)
  │     LiteStore, OllamaEngine, GitAdapter, KnowledgeEntry
  └─→ RemoteClient (httpx)                 → REST API (remote mode)
        Agent sessions, merge, MCP — server-only features
```

**Scope:**
- **Embedded mode (PyO3):** `Workspace.open()`, `store.search()`, `engine.embed()`, `adapter.ingest()` — full local workflow
- **Remote mode (httpx):** Agent sessions, merge coordination, MCP — features that need a running server
- **Async boundary:** PyO3-asyncio bridges Tokio ↔ asyncio for async operations
- Pydantic v2 models mirroring Rust DTOs (typed, auto-validated)
- LangChain `BaseRetriever` subclass — Corvia as drop-in RAG retriever
- Jupyter-native `_repr_html_` for rich display of search results
- Agent session context manager (auto-commit/rollback, remote mode)
- Cross-platform wheels via maturin: manylinux, macOS arm64/x86, Windows
- License: **Apache 2.0** (per D24 — client SDKs are permissive)

**Usage:**

```python
from corvia import Workspace, RemoteClient

# Embedded mode — no server needed
ws = Workspace.open("/path/to/workspace")
ws.ingest("./repos/backend")
results = ws.search("caching strategy", scope="backend")

# Remote mode — for agent coordination, merge, MCP
client = RemoteClient("http://localhost:3000")
async with client.session(agent_name="indexer", scope="backend") as session:
    session.write("New finding about caching")
    # auto-commits on exit

# LangChain integration
from corvia.integrations import CorviaRetriever
retriever = CorviaRetriever(workspace=ws, scope="backend")
```

**Build tooling:** maturin (standard for PyO3 projects — used by polars, ruff, pydantic-core).

**Alternatives considered:**
1. ~~Pure Python httpx client~~ — Misses the opportunity; every AI tool has an HTTP client. PyO3 embed is a differentiator (like polars vs pandas)
2. ~~Selective bindings (expose only search/embed)~~ — Good complexity/value ratio but leaves out ingestion, which is a key local workflow. Full embed is worth the build complexity
3. ~~TypeScript SDK only~~ — Misses the dominant AI ecosystem (Python)
4. ~~REST-only (no SDK)~~ — Maximum adoption friction; no competing tool does this

**Known challenges:**
- Cross-platform wheel CI matrix (maturin + GitHub Actions handles this well — proven by polars, ruff)
- Async bridging (Tokio ↔ asyncio) has sharp edges — pyo3-asyncio crate mitigates
- Binary size (~10-20MB per wheel) — acceptable, polars ships 20MB+ wheels
- Ollama must still be running externally for embeddings — the kernel calls it, not embeds it

**Timeline:** SDK begins in M5 (VS Code extension era, when REST API is stable), published to PyPI in M7 (OSS launch).

### D54: Eval Framework — M6

**Decision:** Implement Corvia's eval/benchmark system (M6 milestone) in Python as the primary language.

**Why Python is superior here:** Eval is inherently iterative — tweak metrics, re-run, visualize, tweak again. Rust's compile cycle kills this workflow. Python's `pandas` + `matplotlib` + `pytest` + `datasets` make eval harnesses trivial. No Rust eval framework exists at comparable maturity. Notebook-driven exploration of results is essential for tuning quality thresholds.

**Scope:**
- Retrieval quality benchmarks: precision@k, recall@k, MRR, NDCG
- Comparison baselines: raw vector search, naive keyword search, competing systems
- LLM-merge quality evaluation: does merged content preserve accuracy?
- Visual reports: heatmaps, score distributions, regression tracking
- Runnable via `corvia eval` CLI or `pytest` — hits REST API
- Jupyter notebooks for interactive result exploration

**Alternatives considered:**
1. ~~Rust eval binary~~ — Compile-time iteration penalty makes metric tuning painful; no mature eval libraries in Rust
2. ~~Shell scripts + jq~~ — Unmaintainable beyond trivial benchmarks; no visualization
3. ~~External eval platform (Weights & Biases, Neptune)~~ — Adds dependency; Corvia should dogfood its own quality measurement

**Timeline:** M6 milestone — this *is* M6, implemented in Python.

### D55: Fine-Tuning Pipeline for Custom Embeddings — Post-M6

**Decision:** Build a Python pipeline for fine-tuning domain-specific embedding models using Corvia's accumulated search/retrieval data.

**Why Python is superior here:** PyTorch and HuggingFace `sentence-transformers` are Python-only at production quality. Fine-tuning embedding models has **zero Rust equivalent**. This is the most absolute language advantage of the three.

**Scope:**
- Collect training pairs from Corvia search logs (query → relevant entry)
- Fine-tune base embedding model (e.g., nomic-embed-text) on domain data
- Evaluate embedding quality before/after on held-out set
- Export model for Ollama/vLLM deployment
- Gated by eval metrics from D54 — only fine-tune when eval shows it's worth it

**Alternatives considered:**
1. ~~Rust bindings to libtorch~~ — Immature, poor HuggingFace ecosystem access, painful debugging
2. ~~Skip fine-tuning entirely~~ — Leaves quality improvement on the table for orgs with domain-specific language
3. ~~External fine-tuning service (OpenAI, Cohere)~~ — Vendor lock-in, can't run locally, violates zero-Docker-first ethos

**Prerequisites:** D54 eval framework must exist first — eval tells you *when* fine-tuning is worth doing and measures the improvement.

**Timeline:** Post-M6, when sufficient search data exists to make fine-tuning meaningful.

**Rust is the kernel. Python embeds Rust; Rust never depends on Python.**

```
┌─────────────────────────────────────────────────────────────┐
│  Python layer (Apache 2.0)                                  │
│  ┌──────────────┐  ┌──────────────┐  ┌───────────────────┐  │
│  │ corvia-py    │  │ corvia-eval  │  │ corvia-finetune   │  │
│  │ (D53, PyO3)  │  │ (D54, pure)  │  │ (D55, pure)       │  │

│         │ PyO3 FFI        │ HTTP              │ HTTP         │
└─────────┼─────────────────┼───────────────────┼──────────────┘
          ▼                 ▼                   ▼
```

- **corvia-py (D53):** Embeds the Rust kernel via PyO3. The `corvia-python` crate is a thin PyO3 translation layer (~1000 LOC) that wraps kernel types. Python calls into Rust for local operations (search, ingest, embed). Remote operations (agent sessions, merge) go over HTTP.
- **corvia-eval (D54) and corvia-finetune (D55):** Pure Python. Connect via REST API (or via corvia-py's embedded mode). No PyO3 dependency — keeps the ML/eval stack simple and iteratable.
- **Rust never imports Python.** The `corvia-python` crate depends on `corvia-kernel`, not the other way around. The kernel compiles and runs without Python installed.

## M3 Scoping Decisions (D56-D59)

*Session: 2026-03-02 (M3 design brainstorm)*

**Context:** With M2.1 complete, M3 ("Knowledge that remembers — and reasons") is next. These decisions resolve the open questions about M3 scope, reasoning approach, and graph population.

### D56: M3 as Single Milestone

**Decision:** Implement temporal, graph, and reasoning as one milestone (no sub-milestones).

Temporal, graph, and reasoning are tightly coupled — reasoning depends on both temporal and graph queries, and the graph needs temporal awareness for `supersedes` chains. Splitting would create artificial boundaries.

**Alternatives considered:**
1. ~~M3.1 (temporal+graph) / M3.2 (reasoning)~~ — Natural split but forces two release cycles for interdependent features
2. ~~M3.1 / M3.2 / M3.3 (one per pillar)~~ — Too granular, adds coordination overhead for tightly coupled work

### D57: Progressive Reasoning — Algorithmic Core, LLM Opt-In

**Decision:** Level 2 reasoning is purely algorithmic (graph traversal + temporal queries). Level 3 findings generation requires an LLM and is opt-in, disabled by default.

**Rationale:** The primary user (coding with Claude subscription) can't provide an API key for reasoning-time LLM calls. Local models via Ollama may lack accuracy for semantic analysis. The temporal+graph storage layer is immediately valuable without LLM — "what changed?", "what depends on this?", "show history" are pure data queries. Algorithmic reasoning (stale entries, dangling references, supersession chain breaks) needs no LLM. LLM-powered findings (semantic gap detection, natural language pattern descriptions) are additive and can use whatever model is available when configured.

**Progressive tiers:**
1. **No LLM needed:** Temporal queries, graph traversal, `corvia history`, `corvia graph`
2. **Algorithmic reasoning:** Stale entry detection, broken supersession chains, orphaned graph nodes, cycle detection
3. **LLM opt-in (Level 3):** Semantic gap analysis, contradiction detection, natural language findings. Requires `[reasoning]` section in `corvia.toml` with model config.

**Alternatives considered:**
1. ~~LLM-required for all reasoning~~ — Blocks core functionality on model availability
2. ~~Defer all reasoning to later milestone~~ — Misses the opportunity to ship useful algorithmic analysis with the storage layer

### D58: Auto-Extract + Agent-Created Graph Edges

**Decision:** Graph edges are populated from two sources: automatic extraction during tree-sitter ingestion, and explicit creation by agents/users via API.

**Auto-extracted relations (during `corvia ingest`):**
- `imports` — file A imports/uses file B (from import statements)
- `calls` — function A calls function B (from call expressions)
- `implements` — type implements trait/interface (from impl blocks)
- `contains` — module contains submodule/function/type (from AST hierarchy)

**Agent-created relations (via `corvia relate` CLI or REST API):**
- `depends_on` — semantic dependency identified by reasoning
- `supersedes` — knowledge entry replaces another (already in KnowledgeEntry.superseded_by)
- `contradicts` — entries make conflicting claims
- `relates_to` — general semantic relationship

**Extensible:** Relation types are strings, not enums. The above are conventions, not a closed set.

**Alternatives considered:**
1. ~~Manual/API only~~ — Graph starts empty, low value until agents populate it
2. ~~Auto-extract only~~ — Misses semantic relationships that only agents can identify

### D59: `corvia upgrade` in M3

**Decision:** Include the LiteStore → FullStore migration command in M3.

Even though the primary development path is LiteStore (zero Docker), having the upgrade path ready means early adopters aren't locked into a storage tier. The command reads `.corvia/knowledge/**/*.json`, bulk-inserts into SurrealDB, migrates graph edges and temporal indexes, and updates `corvia.toml` from `store_type = "lite"` to `store_type = "surrealdb"`.

**Alternatives considered:**
1. ~~Defer to M4 (observability)~~ — M4 already has two-tier focus, but migration is a prerequisite for users who want to upgrade
2. ~~Defer to M7 (OSS launch)~~ — Too late; early testers need the escape hatch

### D61: RAG Pipeline — Trait-Based Staged Architecture (M3.2)

**Decision:** Build the RAG pipeline as three pluggable trait stages in `corvia-kernel`: `Retriever` → `Augmenter` → `GenerationEngine`. A `RagPipeline` struct orchestrates all three. Two modes: `context()` (R+A only, caller brings own LLM) and `ask()` (full R+A+G).

The trait-based design enables benchmarking different strategies (e.g., `VectorRetriever` vs `GraphExpandRetriever`) and adapting to a fast-moving industry. M6 eval harness plugs directly into these traits.

Ships with concrete implementations: `VectorRetriever` (pure vector search), `GraphExpandRetriever` (vector + graph edge expansion with configurable α blending), `StructuredAugmenter` (token-budget-aware prompt formatting with source citations).

**Alternatives considered:**
1. ~~Monolithic RagPipeline (no traits)~~ — Simpler but untestable in isolation, no strategy swapping for benchmarks
2. ~~Layered concrete structs (upgrade ContextBuilder + new RagPipeline)~~ — Clean but retrieval strategy locked; extracting traits later is harder than starting with them
3. ~~Single Retriever trait only~~ — Good middle ground but augmentation may also vary (few-shot vs zero-shot vs structured); full traits future-proof for M6 eval

---

### D62: RAG Observability — D23/D45-Aligned Dual-Layer Telemetry

**Decision:** RAG pipeline observability follows D23 (OpenTelemetry stack) and D45 Part 8 (emit from day one, M4 configures export). Two layers:

- **Layer A (runtime):** `tracing::info_span!` with `corvia.rag.*` naming convention + `metrics::histogram!/counter!` for aggregation. Span tree: `corvia.rag.query` → `corvia.rag.retrieve` → `corvia.rag.augment` → `corvia.rag.generate`.
- **Layer B (eval):** `PipelineTrace` struct returned with every `RagResponse`, capturing per-stage latency, result counts, token usage, and strategy names. Feeds M6 precision@k / MRR / NDCG calculations.

Each trait exposes `fn name(&self) -> &str` for metrics attribution across strategies.

**Alternatives considered:**
1. ~~Custom metrics structs only (no OTel spans)~~ — Breaks D45 convention; M4 Grafana dashboards wouldn't see RAG pipeline
2. ~~OTel spans only (no PipelineTrace return type)~~ — Sufficient for runtime but M6 eval needs programmatic access, not log parsing

### D63: GenerationEngine Trait Name (revises D60 ChatEngine)

**Decision:** The Rust trait for text generation is named `GenerationEngine`, not `ChatEngine`. The D60 proto service stays `ChatService` (reflects wire protocol — chat-completions format). The Rust trait reflects the capability (generate text from prompt), not the protocol.

**Alternatives considered:**
1. ~~Keep ChatEngine~~ — Matches D60 as written, but misleading in RAG context (implies multi-turn conversation)
2. ~~CompletionEngine~~ — Neutral but too tied to "completions" API naming convention, which may not persist

### D64: RAG Token Budget — Model-Aware Auto-Sizing with Override

**Decision:** Context assembly uses model-aware auto-sizing by default: query `GenerationEngine` for context window size, reserve 20% for answer generation, fill remaining with sources by score. Manual override via `max_context_tokens` in `[rag]` config section (0 = auto).

Token estimation uses chars/4 heuristic initially; proper tokenizer can be swapped via `Augmenter` trait.

**Alternatives considered:**
1. ~~Fixed configurable budget only~~ — Requires manual tuning per model, easy to get wrong
2. ~~No budget management~~ — Risks silent truncation or context window overflow

### D65: Chunking Architecture — Template Method Trait with Kernel Pipeline

**Decision:** Chunking uses a `ChunkingStrategy` trait (template method pattern). Adapters implement domain-specific chunking (AST for code, heading-based for markdown). The kernel's `ChunkingPipeline` orchestrates and enforces universal concerns: token budget, merging small chunks, splitting oversized chunks. Adapters control every domain decision via trait methods; kernel ensures compliance. The `chunk()` method returns `ChunkResult` — a wrapper containing both `Vec<RawChunk>` (chunks) and `Vec<ChunkRelation>` (cross-chunk/cross-file relations discovered during chunking). Relations use stable `(source_file, start_line)` identifiers rather than chunk indices, so they remain valid through the pipeline's merge/split steps. Code-aware strategies (e.g., AstChunker) populate relations; simple strategies return an empty relations vec.

**Alternatives considered:**
1. ~~Kernel post-processing only (Approach A)~~ — Kernel can't make domain-aware split decisions, would cut mid-function
2. ~~Kernel owns all chunking (Approach B)~~ — Loses domain knowledge, heavy deps (tree-sitter) move into kernel
3. ~~Adapter-owned with kernel utilities (Approach C)~~ — No enforcement, sloppy adapters can skip utilities

---

### D66: Format Routing — Hybrid Auto-Detect with Adapter Overrides

**Decision:** `FormatRegistry` maps file extensions to `ChunkingStrategy` implementations. Kernel ships defaults for common formats (markdown, config, PDF, fallback). Adapters register strategies for their domain formats (e.g., AST chunking for code extensions) and can override kernel defaults (e.g., a legal adapter replaces generic PdfChunker with clause-aware one). Resolution order: adapter override > adapter register > kernel default > fallback.

**Alternatives considered:**
1. ~~Adapter declares full registry~~ — Adapters must register everything, no out-of-box experience
2. ~~Convention-based auto-detect only~~ — No adapter customization, can't override for domain needs

---

### D67: Sibling Merging — Kernel Default with Strategy Override

**Decision:** After domain chunking, small adjacent chunks merge up to token budget. Kernel provides default merge (adjacent concatenation). Strategies can override via `merge_small()` for domain-aware merging (e.g., AST chunker merges siblings within same `impl` block only). Strategies return `None` to use kernel default, `Some(merged)` to take control.

**Alternatives considered:**
1. ~~Per-strategy only~~ — Duplicates merge logic across every strategy
2. ~~Kernel only, no override~~ — Merges semantically unrelated chunks (two functions from different modules)
3. ~~Defer merging~~ — Industry data (Cursor, cAST) shows merging is critical for chunk quality

---

### D68: M3.3 Scope — Full Format Coverage

**Decision:** M3.3 ships the complete chunking framework with five strategy implementations: AstChunker (code, in adapter-git), MarkdownChunker (prose), ConfigChunker (TOML/YAML/JSON), PdfChunker, and FallbackChunker (recursive text splitting for unknown formats). Full coverage makes this a "tell" feature — demonstrates the abstraction's value across all content types in a repo.

**Alternatives considered:**
1. ~~Framework + code + markdown only~~ — Proves architecture but incomplete format story
2. ~~Framework only with AstChunker~~ — Minimal scope but delays the value proposition

---

### D69: IngestionAdapter Trait — Clean Break, No Migration

**Decision:** `IngestionAdapter` trait changes directly: `ingest()` is replaced by `ingest_sources() -> Vec<SourceFile>` (raw files) + `register_chunking()` (strategy registration). No backward-compatible migration path — project is pre-release with one adapter (GitAdapter). Chunking responsibility moves from adapter output to kernel's ChunkingPipeline.

**Alternatives considered:**
1. ~~Backward-compatible dual path~~ — Unnecessary complexity for pre-release project with one adapter
2. ~~Keep adapter-owned chunking~~ — Misses the opportunity to standardize the pipeline
