## Overview

M2.1 adds the **workspace** concept to Corvia — a purpose-built directory for multi-repo knowledge management — and delivers a **devcontainer-based demo** that showcases Corvia working across multiple repos. The workspace serves three roles: template for users, 1-click demo for evaluators, and the development environment we use to build Corvia M3+.

## Architecture

### Workspace Model (D46)

A workspace is a purpose-built directory. Target repos are NOT polluted with Corvia artifacts.

```
my-workspace/
├── corvia.toml              ← has [workspace] section
├── .corvia/                 ← knowledge store, indexes, coordination
│   ├── .git/                ← knowledge git (source of truth)
│   ├── knowledge/           ← git-tracked JSON knowledge files
│   ├── hnsw_index/          ← LiteStore vector index
│   ├── coordination.redb
│   └── .gitignore
├── repos/                   ← cloned or symlinked target repos
│   ├── repo-a/
│   └── repo-b/
└── .devcontainer/           ← optional
```

### Config Format (D47)

Unified `corvia.toml` with optional `[workspace]` section (Cargo-style):

```toml
[project]
name = "my-project"
scope_id = "my-org"

[workspace]
repos_dir = "repos"

[[workspace.repos]]
name = "backend"
url = "https://github.com/org/backend"
local = "/home/dev/backend"           # optional local override
namespace = "backend"

[[workspace.repos]]
name = "frontend"
url = "https://github.com/org/frontend"
namespace = "frontend"

[storage]
store_type = "lite"
data_dir = ".corvia"

[embedding]
provider = "ollama"
model = "nomic-embed-text"
url = "http://127.0.0.1:11434"
dimensions = 768

[server]
host = "127.0.0.1"
port = 8020
```

**Detection**: If `corvia.toml` contains `[workspace]`, the directory is a workspace. If not, backward-compatible single-project mode.

**Namespace mapping**: Each repo's `namespace` field maps to the `workstream` segment: `{org}:{scope_id}:{namespace}:{source}:{version_ref}`.

**Local overrides**: If `local` path exists and is a valid git repo, use it directly (no clone). Otherwise clone from `url` into `{repos_dir}/{name}`.

## CLI Surface (D48)

### New Subcommand Group

```
corvia workspace create <name>     Scaffold a new workspace
    --repo <url>                   Add a repo (repeatable)
    --template <url|path>          Scaffold from a template workspace
    --local <name>=<path>          Set local override for a repo

corvia workspace init              Initialize workspace in current directory
                                   (reads existing corvia.toml, clones missing repos, sets up .corvia/)

corvia workspace status            Show repos, ingest state, knowledge counts per namespace

corvia workspace add <url>         Add a repo to the workspace
    --name <name>                  Override repo name (default: derived from URL)
    --namespace <ns>               Override namespace (default: same as name)
    --local <path>                 Set local path override

corvia workspace remove <name>     Remove a repo from workspace config
    --purge                        Also delete cloned repo + its knowledge entries

corvia workspace ingest            Ingest all workspace repos
    --fresh                        Re-ingest from scratch (discard existing knowledge)
    --repo <name>                  Only ingest this specific repo
```

### Existing Command Adaptation

- `corvia ingest` (no path) in a workspace → equivalent to `corvia workspace ingest`
- `corvia search` → results include namespace (repo origin) in output
- `corvia status` → shows workspace-level info when in a workspace
- `corvia init` → if `corvia.toml` with `[workspace]` exists, behaves as `corvia workspace init`

### What `corvia workspace create` Does

1. Creates `<name>/` directory
2. Generates `corvia.toml` with `[workspace]` section from `--repo` flags
3. If `--template` provided, clones template and adapts config
4. Runs `corvia workspace init` (clones repos, initializes `.corvia/`)
5. Runs first ingest of all repos

## Demo: `corvia-workspace` Repository (D50, D51, D52)

### Triple Role

| Role | Description |
|------|-------------|
| **Template** | `corvia workspace create --template corvia-workspace` scaffolds new workspaces |
| **Demo** | Clone + open in Codespaces/DevPod → instant working Corvia with real multi-repo knowledge |
| **Dev environment** | The workspace we use to develop Corvia M3+. Knowledge accumulates as we build. |

### Repository Structure

```
corvia-workspace/
├── corvia.toml                      ← [workspace] declaring corvia + adapter
├── .corvia/                         ← pre-ingested knowledge (checked in)
│   ├── .git/                        ← knowledge git history
│   ├── knowledge/                   ← JSON entries for both repos
│   └── hnsw_index/                  ← pre-built vector index
├── repos/                           ← .gitignore'd (cloned on init)
│   └── .gitkeep
├── .devcontainer/
│   ├── devcontainer.json
│   ├── Dockerfile                   ← Corvia binary + Ollama
│   └── scripts/
│       ├── post-create.sh           ← clone repos, pull Ollama model
│       └── post-start.sh            ← corvia serve --mcp &
├── .gitignore                       ← repos/, .corvia/coordination.redb, ephemeral files
└── README.md                        ← "Open in Codespaces" badge, quickstart
```

### Demo Config

```toml
[project]
name = "corvia-demo"
scope_id = "corvia"

[workspace]
repos_dir = "repos"

[[workspace.repos]]
name = "corvia"
url = "https://github.com/<org>/corvia"
namespace = "kernel"

[[workspace.repos]]
name = "corvia-adapter-git"
url = "https://github.com/<org>/corvia-adapter-git"
namespace = "adapter"

[storage]
store_type = "lite"
data_dir = ".corvia"

[embedding]
provider = "ollama"
model = "nomic-embed-text"
url = "http://127.0.0.1:11434"
dimensions = 768

[server]
host = "127.0.0.1"
port = 8020
```

### Devcontainer

```jsonc
{
  "name": "Corvia Demo Workspace",
  "build": { "dockerfile": "Dockerfile" },
  "postCreateCommand": ".devcontainer/scripts/post-create.sh",
  "postStartCommand": ".devcontainer/scripts/post-start.sh",
  "forwardPorts": [8020],
  "customizations": {
    "vscode": {
      "extensions": [],
      "settings": {
        "corvia.serverUrl": "http://localhost:8020"
      }
    }
  }
}
```

**Lifecycle**:
- `postCreate`: `corvia workspace init` (clones repos, pulls Ollama model if missing)
- `postStart`: `corvia serve --mcp &` (background server)
- Pre-ingested knowledge means instant search without waiting for embedding

### Pre-Ingested vs. Fresh

- **Default**: Ships with pre-ingested `.corvia/knowledge/` + `hnsw_index/`. Instant demo.
- **`--fresh`**: `corvia workspace ingest --fresh` rebuilds from scratch. User choice.
- **Release process**: CI regenerates pre-ingested knowledge per release. Checkout repos at release tag → ingest → commit updated knowledge → tag.

### Two Operating Modes

```
Mode 1 (native):       corvia workspace init → local Ollama → develop normally
Mode 2 (devcontainer):  open in Codespaces → Ollama in container → same workspace, portable
```

The workspace works identically in both. Devcontainer is optional packaging, not required.

## What the Demo Showcases

Before M5 (VS Code extension), the demo is CLI-driven:

- `corvia search "IngestionAdapter"` → finds trait in kernel + impl in adapter (cross-repo)
- `corvia search "how does chunking work"` → surfaces tree-sitter logic from adapter
- `corvia workspace status` → shows both repos, entry counts, namespace breakdown
- `corvia agent list` → agent coordination via REST/MCP
- Cross-repo knowledge queries that span both namespaces

## Implementation Approach (Approach C — Parallel Tracks)

**Keystone**: Define the `[workspace]` config format first — it's the contract both tracks depend on.

**Stream 1 — Workspace CLI** (in `corvia` monorepo):
1. Parse `[workspace]` section in config
2. `corvia workspace create` (scaffold + generate config)
3. `corvia workspace init` (clone repos, set up `.corvia/`)
4. `corvia workspace add/remove` (config manipulation)
5. `corvia workspace ingest` (multi-repo ingest with namespace isolation)
6. `corvia workspace status` (workspace-aware status display)
7. Adapt existing commands (`ingest`, `search`, `status`) for workspace mode

**Stream 2 — Demo Repo** (`corvia-workspace` repo):
1. Create repo structure (directory layout, `.gitignore`, README)
2. Write `corvia.toml` with workspace config for corvia + adapter
3. Devcontainer: Dockerfile (Corvia + Ollama), lifecycle scripts
4. Pre-ingest knowledge from both repos
5. Validate: clone → open devcontainer → search works immediately

**Convergence**: Stream 2 exercises Stream 1's commands as they're delivered. If the demo doesn't work, the CLI isn't done.
